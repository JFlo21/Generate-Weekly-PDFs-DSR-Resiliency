## Warnings

### WR-01: Cross-sheet WR overlap deletes / prunes legitimate non-subcontractor helper+primary artifacts (data loss bypassing `valid_wr_weeks`)

**File:** `generate_weekly_pdfs.py:2750-2757` (TARGET cleanup gate),
`generate_weekly_pdfs.py:3162-3166` (hash-prune helper clause);
scope built at `generate_weekly_pdfs.py:3052-3072`

**Issue:**
`_build_subcontractor_wr_scope` derives the in-scope WR set purely from the
presence of a `_REDUCEDSUB` group key — it keys on **WR number only**, with
**no sheet-origin (`__source_sheet_id`) discrimination**. Both new consumers
then treat *every* `helper`/`primary` artifact for an in-scope WR as
off-contract:

- TARGET cleanup (L2750-2757):
  `wr in sub_wr_scope and variant in {'helper','primary'}` appends to
  `off_contract_attachments` and `continue`s **before**
  `identity_groups[ident].append(att)` (L2758). The attachment therefore never
  reaches the `valid_wr_weeks` keep-newest logic (L2780-2782) — it is deleted
  **unconditionally, bypassing both `valid_wr_weeks` and `KEEP_HISTORICAL_WEEKS`.**
- Hash-prune (L3162-3166):
  `_hk_wr in _sub_wr_scope and (... or _hk_variant == 'helper')` drops the
  hash-history entry for that WR.

The pipeline groups globally across all source sheets and decides
`is_subcontractor_row` **per-row** by `__source_sheet_id` (L4665-4669), so a
single WR# can legitimately carry rows on BOTH a subcontractor sheet (emitting
`_REDUCEDSUB` → in scope) AND a non-subcontractor sheet (emitting a genuine,
currently-shipping `('helper', ...)` group → a live `_Helper_<name>.xlsx` on
TARGET_SHEET_ID, protected by `valid_wr_weeks`). For such a mixed WR:

1. **Deletion of a live attachment + delete/re-upload churn.** Cleanup runs
   *after* the upload phase (L7920-7947), so the freshly re-uploaded legitimate
   non-sub `_Helper_<name>.xlsx` (and any bare-`primary` non-sub file) is
   deleted on the same run. The one-time hash-prune also drops its hash entry,
   so next run it regenerates, re-uploads, and is deleted again — a permanent
   delete/re-upload loop that leaves the legitimate non-sub helper attachment
   *absent* from TARGET between cleanup and the next run's upload. That is
   operator-visible loss of a valid billing artifact, not just the intended
   pre-fix-orphan cleanup.
2. **The risk was half-considered, then unguarded.** The `[2026-05-19 22:00]`
   ledger entry's "WR-sharing prune edge case" only reasoned about pruning the
   in-scope WR *correctly* (treating overlap as a feature: "still gets pruned
   correctly"). The inverse — a WR whose `helper`/`primary` entry is a **live
   non-subcontractor** artifact that must be preserved — was not considered, so
   this path has no guard.

Realistic numeric subcontractor WRs that never appear on a non-sub sheet are
unaffected, which is why the single-sheet synthetic test corpus and the UAT run
did not surface it. Likelihood is bounded to the WR-overlap case, but the
consequence (silent deletion of a valid, currently-generated attachment that
bypasses `valid_wr_weeks`) is a production-safety regression in the highest-risk
cleanup path on a production billing pipeline.

**Fix:**
Discriminate by live-identity (or by sheet origin) so only genuinely
off-contract artifacts are removed. Preferred — exempt any attachment whose
parsed identity is in `valid_wr_weeks` (i.e. it was generated/validated this
run), since a live current-run identity must never be deleted:

```python
if (
    sub_wr_scope is not None
    and wr in sub_wr_scope
    and sub_offcontract_variants is not None
    and variant in sub_offcontract_variants
    and ident not in valid_wr_weeks   # never delete a live, current-run identity
):
    off_contract_attachments.append(att)
    continue
```

Apply the symmetric guard in the hash-prune: before appending a `helper`/
`primary` key to `_orphans_to_drop`, skip it if the same WR/week/variant has a
live group in `groups` this run (i.e. reconstruct the `current_keys`-style key
and check membership). Add a regression test for a mixed WR — rows on both
`_SUB_SHEET_ID` and a non-sub sheet — asserting the non-sub `_Helper_<name>`
attachment and its hash entry survive both the cleanup and the prune.

## Info

### IN-01: Hash-prune `helper` clause is broader than its own documented 6-part shape

**File:** `generate_weekly_pdfs.py:3162-3166`

**Issue:**
The orphan condition is
`_hk_wr in _sub_wr_scope and ((len(_parts) == 4 and _hk_variant == 'primary' and _parts[3] == '') or _hk_variant == 'helper')`.
The `or _hk_variant == 'helper'` clause carries no part-count or
blank-identifier constraint, while the primary clause is tightly bounded. The
inline comment (L3143-3147, L3155-3161) and the function docstring describe
helper keys as exactly "6 parts (`wr|week|helper|foreman|dept|job`)", but the
code drops a `helper`-variant key of *any* part count (4, 5, 6, 7+) for an
in-scope WR. This is harmless today — subcontractor `helper` variants are being
fully retired and the version-gate makes it one-time — so it is not a
behavioural defect now, but the clause is broader than its documentation, which
invites drift if a future `helper`-shaped variant key with a different part
count is introduced. It also compounds the WR-01 mixed-WR concern.

**Fix:**
Align the comment to the actual behaviour (any `helper`-variant key with
`len >= 4` is dropped), or, when fixing WR-01, fold the live-entry exemption in
here too so the broadening can never drop a legitimate helper key. No code
change is strictly required for this phase if WR-01 is addressed.
