# Phase 1.1: Subcontractor Helper-Shadow Rescue — Code Review Report

**Reviewed:** 2026-05-19T00:00:00Z
**Depth:** standard
**Files Reviewed:** 10
**Status:** issues_found

## Summary

Reviewed the Phase 1.1 implementation that closes four subcontractor
helper-shadow bugs (Bug A pre-acceptance rescue; Bug B1 variant
partitioning cascade; Bug B2 PPP cleanup whitelist; Bug C
`billing_audit.lookup_attribution` claim-history reader) plus the
SUB-12 hash-history prune.

Findings cluster around three classes of defect:

1. **Schema/documentation gaps that the CLAUDE.md
   `[2026-04-25 12:00]` rule explicitly forbids** — the new
   `lookup_attribution` PostgREST RPC is read by the pipeline but
   has no contract entry in `billing_audit/schema.sql`, and the
   `billing_audit/writer.py` module docstring still claims "No
   read / hydration path." Both directly violate the data-contract
   discipline that the Living Ledger was created to enforce.
2. **Bug A coverage is structurally hollow** — `_subcontractor_rescue_price`
   gets four unit tests, but the rescue *branch* inside
   `_fetch_and_process_sheet` has only a source-grep guard. The
   `tests/test_subcontractor_helper_shadow_rescue.py` author
   acknowledges this is the "mirror passes but production reverted"
   trap and writes a Living Ledger-style preamble warning against
   it — then ships a test file that does exactly that for Bug A.
3. **Data-flow drift in the rescue diagnostic + Bug C empty-helper
   handling** — small but real correctness gaps where operator-
   visible identifiers can carry path-traversal characters into log
   bodies and where a whitespace-only frozen helper string flows
   into a malformed group-key prefix.

The four kill-switch env vars, the workflow pin, the per-WR dedup
WARNING, the off-contract delete loop, and the hash-prune sentinel
all look correct in structure and survive the existing test corpus.
The findings below are surgical — none require a Phase 1.1 redesign,
but the two BLOCKERs must close before this merges to master.

## Blockers

### CR-01: `lookup_attribution` PostgREST RPC has NO contract entry in `billing_audit/schema.sql`

**File:** `billing_audit/schema.sql` (entire file — no entry)
**Cross-ref:** `billing_audit/writer.py:650-756`, `billing_audit/writer.py:699`,
`generate_weekly_pdfs.py:4790`

**Issue:**
Phase 1.1 ships `lookup_attribution(wr, week_ending, smartsheet_row_id)`
as a new RPC dependency. The pipeline calls it per qualifying
helper row. Per CLAUDE.md `[2026-04-25 12:00]` Living Ledger
entry rule 1 (verbatim): *"Any new Supabase table or column the
pipeline reads/writes MUST be defined in `billing_audit/schema.sql`
in the same PR that adds the Python code. The repo cannot have a
writer whose matching DDL exists only in a Supabase Dashboard
somebody else's hands."*

`schema.sql` documents the `freeze_attribution` RPC parameter
contract (L128-156) but contains no entry for `lookup_attribution`,
no DDL for `billing_audit.attribution_snapshot` (the table the new
RPC reads from), and no return-shape declaration for the dict
`{helper, helper_dept, source_run_id}` the Python reader expects.
The reader's own docstring (writer.py:699) tells callers to look at
`schema.sql` for the contract — but the contract is not there.

This is the same failure mode the `[2026-04-25 12:00]` ledger
entry was created to prevent: a Python writer/reader whose matching
DDL exists "only in a Supabase Dashboard." When (not if) the data
team rotates or rebuilds the Supabase project, the contract is
lost and PGRST106/PGRST404 storms the pipeline.

The runtime side-effect is bounded by Bug C's run-global kill
switch (a missing RPC trips PGRST106 → `_global_disable_reason`
flips → all subsequent calls return None → rows fall back to
current helper), so the pipeline does NOT crash. But the
operator-visible signal of *what to deploy* is absent from the
repository.

**Fix:**
Add a `lookup_attribution` contract block to
`billing_audit/schema.sql` immediately after the existing
`freeze_attribution` block at L128-156. Use the same comment-only
style (the data team owns the function body in the Supabase
Dashboard; the SQL file documents the parameter + return shape).
Minimal addition:

```sql
-- ── attribution_snapshot (READ surface) ─────────────────────
-- ``billing_audit.attribution_snapshot`` is the per-row personnel
-- snapshot table populated by ``freeze_attribution``. The Python
-- reader (``billing_audit.writer.lookup_attribution``) reads from
-- it via the ``lookup_attribution`` RPC defined below. PRIMARY
-- KEY shape: (wr, week_ending, smartsheet_row_id). Columns the
-- reader depends on: ``helper TEXT``, ``helper_dept TEXT``,
-- ``source_run_id TEXT``. Owned by the data team; pipeline must
-- tolerate column additions but the three names above are load-
-- bearing for Phase 1.1 Bug C claim-history attribution.

-- ── lookup_attribution (RPC) ────────────────────────────────
-- The ``lookup_attribution`` Postgres function is NOT defined
-- here — its body is deployed and maintained directly in the
-- Supabase project, mirroring the ``freeze_attribution`` pattern
-- above. The pipeline's contract with it is:
--
--   PARAMETERS (all named, p_<name>):
--     p_wr                TEXT
--     p_week_ending       DATE
--     p_smartsheet_row_id BIGINT
--
--   RETURNS: a row (or scalar) with
--     ``helper TEXT, helper_dept TEXT, source_run_id TEXT``,
--     or NULL when no frozen attribution exists for the given
--     (wr, week_ending, smartsheet_row_id) tuple.
--
-- The exact body lives in Supabase. Do NOT rename or change
-- the parameter names without the corresponding update in
-- ``billing_audit/writer.py:lookup_attribution``.
```

Confirm with the data team that this matches the deployed RPC
body before flipping `SUBCONTRACTOR_HELPER_CLAIM_ATTRIBUTION_ENABLED=1`
in production. The 01.1-VALIDATION.md "Manual-Only Verifications"
row already names this coordination — but it MUST be paired with
the SQL block above so future readers can self-serve.

### CR-02: `billing_audit/writer.py` module docstring claims "no read / hydration path" but `lookup_attribution` is now in the file

**File:** `billing_audit/writer.py:1-14`

**Issue:**
The module docstring at the top of `billing_audit/writer.py` reads:

```
"""Supabase attribution snapshot writer (shadow mode).

Writes only. No read / hydration path — that lands in the follow-up
PR. All public functions are silent no-ops when:
...
```

Phase 1.1 added `lookup_attribution` to the same file (L650-756).
The docstring is now factually wrong on two counts:

1. The module is no longer "writes only" — it has a reader.
2. "That lands in the follow-up PR" — the follow-up PR IS this one.

A reader who lands on this file expecting only writers will
miss the new public surface. This is the inverse of the
`billing_audit/__init__.py` docstring (which WAS updated in Plan
01.1-04 Task 2 to acknowledge the read-path arrival). The package
docstring and the module docstring contradict each other today.

Beyond accuracy: the module docstring also says *"NEVER emit
per-row details (WR, foreman, helper, vac_crew names)"* and
*"Only aggregate counter summaries at INFO"* — that discipline
covers the writer functions but is silent on the reader's surface.
A reader contributor reading this docstring may not realize the
reader is the one place where per-row helper names DO leave the
package (as the return-value dict). The docstring should call out
the reader's exception to the no-PII-out rule explicitly.

**Fix:**
Replace the writer.py module docstring with:

```python
"""Supabase attribution snapshot writer + reader (shadow mode).

Writers: ``freeze_row`` upserts per-row personnel into
``attribution_snapshot`` (first-write-wins). ``emit_run_fingerprint``
upserts per-WR-week-variant rows into ``pipeline_run`` for
mid-week-swap drift detection.

Reader (Phase 1.1 / Bug C / SUB-11): ``lookup_attribution``
queries the frozen helper for ONE row via the matching PostgREST
RPC. Unlike the writers, the reader DOES return per-row helper
PII to the caller in its dict return value — that is the
point. Callers MUST sanitize the returned ``helper`` field via
``_RE_SANITIZE_HELPER_NAME`` before embedding it in filenames,
group keys, or any log body, and MUST extend
``_PII_LOG_MARKERS`` in generate_weekly_pdfs.py for any new INFO
log body that quotes the reader's return value.

All public functions are silent no-ops when:
- Supabase credentials are unset or TEST_MODE is on.
- The relevant feature flag in ``billing_audit.feature_flag`` is
  off (writers only; the reader has NO feature-flag gate — its
  kill switch lives at the caller in
  ``SUBCONTRACTOR_HELPER_CLAIM_ATTRIBUTION_ENABLED``).
- A row does not meet the freeze criteria (missing row id, or
  ``Units Completed?`` unchecked).

Logging discipline (writers): NEVER emit per-row details (WR,
foreman, helper, vac_crew names). Only aggregate counter summaries
at INFO. The reader can carry PII in its return value but emits
no logs of its own — log discipline is enforced at the caller.
"""
```

Also explicitly note that the reader bypasses
`billing_audit.feature_flag` (it has no `_flag_enabled_or_unknown`
gate). Today a future operator who reads the writer module
expecting "all public functions respect the feature flag" would
be wrong about `lookup_attribution`.

### WR-02: Whitespace-only `helper` value from `lookup_attribution` becomes a malformed empty-suffix group key

**File:** `generate_weekly_pdfs.py:4840-4847`
**Cross-ref:** `billing_audit/writer.py:751` (reader's helper-truthiness check)

**Issue:**
The reader's helper-truthiness check is:

```python
if isinstance(data, dict):
    return data if data.get("helper") else None
```

`data.get("helper")` is falsy only when `helper` is None, empty
string `""`, `0`, or `False`. It is TRUTHY for a whitespace-only
string like `"   "`. So the reader returns `{'helper': '   '}`
without filtering.

At the caller:

```python
else:
    _frozen_helper = _attribution_row.get('helper')
    if _frozen_helper:                              # "   " is truthy
        _attributed_helper = str(_frozen_helper).strip()  # → ""
    else:
        _attribution_reason = 'no_history'
```

`str("   ").strip()` is `""`. Then downstream:

```python
_helper_sanitized = (
    _RE_SANITIZE_HELPER_NAME.sub('_', _attributed_helper)[:50]
)
rs_helper_key = (
    f"{week_end_for_key}_{wr_key}_REDUCEDSUB_HELPER_"
    f"{_helper_sanitized}"   # = ""
)
```

The resulting group key is `"041926_91467680_REDUCEDSUB_HELPER_"`
— a trailing-underscore variant with NO helper name. The filename
`WR_91467680_WeekEnding_041926_..._ReducedSub_Helper_.xlsx` parses
in `build_group_identity` as `(wr='91467680', week='041926',
variant='reduced_sub_helper', identifier='')`. The Phase 1.1 Bug B2
PPP whitelist filter accepts `reduced_sub_helper` as in-contract,
so this malformed identifier-empty key uploads to PPP and lives
there forever.

The "defense-in-depth" comment at L4844-4846 says *"Reader's
contract already filters empty-string helpers to None; this guard
is defense-in-depth."* That is exactly the assertion this finding
contradicts — the reader filters `""` but NOT `"   "`.

The same hole exists if a future RPC body emits other Python-truthy
edge values: `"None"` (the literal string), `"\t"`, or a
zero-width space character (Unicode U+200B), etc.

**Fix:**
Tighten BOTH the reader (defense at the boundary) AND the caller
(defense in depth) to require a stripped non-empty string:

In `billing_audit/writer.py:lookup_attribution`:

```python
def _has_helper(row: dict) -> bool:
    helper = row.get("helper")
    return isinstance(helper, str) and helper.strip() != ""

data = getattr(result, "data", None)
if isinstance(data, dict):
    return data if _has_helper(data) else None
if isinstance(data, list) and data:
    first = data[0]
    if isinstance(first, dict) and _has_helper(first):
        return first
return None
```

In `generate_weekly_pdfs.py:group_source_rows`:

```python
else:
    _frozen_helper = _attribution_row.get('helper')
    _frozen_helper_clean = (
        str(_frozen_helper).strip() if _frozen_helper else ''
    )
    if _frozen_helper_clean:
        _attributed_helper = _frozen_helper_clean
    else:
        _attribution_reason = 'no_history'
```

The cleaned check at the caller catches the same edge case if the
reader regresses.

### WR-03: `_wr_diag` in the Bug A rescue diagnostic log is NOT sanitized at the derivation site

**File:** `generate_weekly_pdfs.py:4022-4031`

**Issue:**
The Bug A rescue diagnostic INFO log emits the row's
`Work Request #` via:

```python
_wr_diag = (
    str(work_request).split('.')[0]
    if work_request else '<unknown>'
)
logging.info(
    f"💲 Subcontractor pre-acceptance rescue: "
    f"WR={_wr_diag}, "
    f"CU={row_data.get('CU')}, "
    f"rescued=${_rescued:.2f}"
)
```

`work_request` is the raw Smartsheet column value — operator-
controlled. The `.split('.')[0]` only strips the numeric `.0`
decimal suffix that Smartsheet sometimes appends; it does NOT
strip filesystem-unsafe characters. CLAUDE.md `[2026-04-23 12:00]`
ledger entry rule 1 (verbatim):

> "Any user-controllable string (row field, Smartsheet cell value,
> env-var-derived identifier) that flows into `os.path.join(...)` /
> `workbook.save(...)` / any `open(path, 'w')` MUST pass through a
> filesystem-safety sanitizer at each derivation site — not just
> at the final filename assembly — so downstream comparisons
> (history keys, attachment prefix matching) stay consistent.
> Reuse `_RE_SANITIZE_HELPER_NAME` ([^\w\-]) or a tighter pattern;
> do not invent new ones per-site."

This particular `_wr_diag` does NOT flow into a filename, but it
DOES flow into a log body that downstream consumers (Sentry,
log aggregators, ops dashboards) will treat as a stable WR
identifier. A row with `Work Request # = "1234/../evil"` would
log `WR=1234/../evil`. The diagnostic is gated on
`FILTER_DIAGNOSTICS and sheet_row_counter < DEBUG_ESSENTIAL_ROWS`
(small blast radius), and the PII marker `"Subcontractor
pre-acceptance rescue"` covers Sentry Logs forwarding, but the
local log file and any tail-based ops tooling still get the raw
value.

The plan's own NOTE in 01.1-01 Task 2 explicitly called this out:

> "NOTE on `wr_key_for_diag`: if this variable does not yet exist
> in this scope, derive it from the row's `Work Request #` field
> at the top of the rescue branch using the same
> `_RE_SANITIZE_HELPER_NAME.sub('_', str(wr_raw).split('.')[0])[:50]`
> sanitizer pattern from `[2026-04-23 18:25]`."

The implementation skipped the sanitizer.

**Fix:**
Apply the sanitizer at the derivation site:

```python
_wr_diag = (
    _RE_SANITIZE_HELPER_NAME.sub(
        '_', str(work_request).split('.')[0]
    )[:50]
    if work_request else '<unknown>'
)
```

The sanitizer is idempotent (CLAUDE.md `[2026-04-23 18:25]`), so
applying it at this site does not conflict with any downstream
sanitization. Realistic numeric WR# values pass through unchanged.

### WR-04: `_attribution_reason` is initialized OUTSIDE the lookup gate; the variable persists across iterations of the inner else branch

**File:** `generate_weekly_pdfs.py:4782-4783, 4855-4858`

**Issue:**
The Bug C integration block declares `_attribution_reason` near the
top of the helper-shadow construction (L4783):

```python
_attributed_helper = helper_foreman  # D-12 default
_attribution_reason: str | None = None
if (
    is_subcontractor_row
    and SUBCONTRACTOR_HELPER_CLAIM_ATTRIBUTION_ENABLED
):
    # ... reader path sets _attribution_reason to None,
    # 'no_history', or 'fetch_failure' ...
```

When the four-gate condition (`is_subcontractor_row AND
SUBCONTRACTOR_HELPER_CLAIM_ATTRIBUTION_ENABLED AND _valid_helper_row
AND _helper_mode_enabled`) is False, the lookup path is skipped
entirely. `_attribution_reason` stays at its initial value (None).
The downstream WARNING gate at L4855-4858:

```python
if (
    is_subcontractor_row
    and SUBCONTRACTOR_HELPER_CLAIM_ATTRIBUTION_ENABLED
    and _attribution_reason in ('no_history', 'fetch_failure')
):
    ...
```

correctly skips logging when `_attribution_reason is None`. Good.

The subtle issue: the comment ABOVE the reader call (the `else:`
on L4606-4612 in the plan, but in the current source, no such
explicit else exists; the "kill switch off" path was folded into
the absence of the gate). The code reads as if the kill-switch-off
path explicitly sets `_attribution_reason = None` — but it doesn't,
because the entire reader block is wrapped in the four-gate `if`.
For someone debugging "why didn't the WARNING fire for this
subcontractor row?" the only way to know is to step through and
see that one of the four gates was False — there's no per-gate
log line that explains which.

This isn't a correctness bug today. It IS a maintainability
hazard. The plan's `<action>` section (Plan 01.1-04 Step 4) showed
an explicit `else: _attribution_reason = None` branch with the
comment "Kill switch off OR not a subcontractor row — D-15 scope."
That branch is missing from the implementation, so the per-gate
intent is implicit.

**Fix:**
Either (a) add the explicit `else` branch the plan called for,
making the four-gate decision tree readable in the source; or (b)
add a structured DEBUG log inside the reader branch that names the
resolved `_attribution_reason` after each row. Option (a) is
cheaper:

```python
if (
    is_subcontractor_row
    and SUBCONTRACTOR_HELPER_CLAIM_ATTRIBUTION_ENABLED
):
    # ... existing reader path ...
else:
    # Kill switch off OR not a subcontractor row — D-15 scope.
    # Phase 1 behavior (`helper_foreman` flows through
    # unchanged); no read; no WARNING. The kill-switch-off
    # WARNING is suppressed because the operator already chose
    # this fall-back deliberately by flipping the env var.
    _attribution_reason = None  # explicit
```

This also makes the WARNING-gate's `_attribution_reason in
('no_history', 'fetch_failure')` check self-documenting — the
exhaustive truth-table is visible in the source.

### WR-05: `lookup_attribution` does NOT consult the `billing_audit.feature_flag` table — read path bypasses an established kill switch

**File:** `billing_audit/writer.py:719-723`
**Cross-ref:** `billing_audit/writer.py:418` (writer's `_FLAG_WRITE` gate)

**Issue:**
Every other public function in `billing_audit/writer.py` consults
the feature_flag table:

- `freeze_row`: `if not _flag_enabled_or_unknown(_FLAG_WRITE): return False` (L418)
- `emit_run_fingerprint`: `if not _flag_enabled_or_unknown(_FLAG_FINGERPRINT): return` (L547)

`lookup_attribution` does NOT. Its only gates are (a) `get_client()
is None`, and (b) the caller-side env var
`SUBCONTRACTOR_HELPER_CLAIM_ATTRIBUTION_ENABLED`.

That means an operator who flips
`billing_audit.feature_flag.write_attribution_snapshot=false` to
stop the integration (e.g., during a Supabase incident or schema
migration) will stop the WRITES but NOT stop the READS. The
pipeline will still issue per-row `lookup_attribution` RPC calls
that hit the malfunctioning table. Per
`_classify_postgrest_error`, a malformed table emitting SQLSTATE
`42P01` / `42703` lands the integration in the
`_PG_SQLSTATE_PERMANENT_PREFIXES` permanent classifier — but each
row issues its own RPC before the per-op breaker accumulates
enough hits to trip.

The pre-PR review threshold of "feature flag stops the
integration" is silently broken for the read path. The
`billing_audit/__init__.py` docstring even says:

> "Apply [the schema] manually in the Supabase SQL Editor before
> enabling this integration on a new project."

…implying that the feature-flag table IS the master gate. It is
no longer the master gate for `lookup_attribution`.

The Plan 01.1-04 `<action>` Task 1 explicitly said:

> "Do NOT add a new feature flag check inside `lookup_attribution`
> itself. The kill switch lives in `generate_weekly_pdfs.py` at the
> call site (Task 3) — keeping the reader gate-free preserves
> testability and lets future callers reuse the reader for other
> purposes (audit queries, debug tools)."

That argument is defensible for unit-test ergonomics but does
NOT explain why operators should be denied a second kill switch
that already exists in the project. The Bug C env-var kill is a
code-level switch (requires workflow change); the feature_flag
kill is a runtime-flippable switch (Supabase UI).

**Fix:**
Add a new feature flag in `billing_audit/feature_flag` —
`read_attribution_snapshot` — and gate `lookup_attribution` on it
using the same `_flag_enabled_or_unknown` fail-open pattern as
the writers. Defaulted to FALSE in `schema.sql` (alongside
`write_attribution_snapshot`) so a new environment must opt in.
Document in `website/docs/reference/environment.md`'s Bug C
section that operators now have TWO kill switches: the
`SUBCONTRACTOR_HELPER_CLAIM_ATTRIBUTION_ENABLED` env var
(code-reviewable through workflow) and the
`read_attribution_snapshot` flag (runtime-flippable via Supabase
UI), and document the precedence (both must be on for the reader
to fire).

If keeping a gate-free reader is intentional for ergonomics, then
make that decision explicit by extending the writer.py module
docstring (WR-CR-02 already requires this) to call out that the
reader deliberately bypasses the feature_flag table — and update
the runbook so on-call doesn't reach for the wrong switch during
an incident.

> `RES_GROUPING_MODE='both'`; non-production primary-only mode
> preserves legacy behavior."

The implementation correctly leaves the `'primary'` branch
unchanged (L4594-4597). Good.

However, a subcontractor row processed under
`RES_GROUPING_MODE='primary'` STILL emits both the legacy primary
key AND the new `_REDUCEDSUB` / `_AEPBILLABLE` variant keys (the
variant emission block at L4687 fires regardless of grouping
mode). For the production `'both'` mode this is fine because the
B1 gate suppresses the legacy primary. For `'primary'` mode the
legacy primary AND the variants coexist — the same "duplicate
file" failure mode Bug B1 was created to close.

Two scenarios where this matters:

1. **A developer or ad-hoc operator running locally with
   `RES_GROUPING_MODE=primary`** (e.g., `MAX_GROUPS=5
   RES_GROUPING_MODE=primary python generate_weekly_pdfs.py` for
   a quick check) will get DIFFERENT output than the production
   `both` mode — the local dev run sees the bug; production does
   not. Silent divergence.
2. **A future plan that sets `RES_GROUPING_MODE='primary'`
   experimentally** would silently regress Bug B1's closure on
   subcontractor sheets.

The implementation comment at L4602-4609 says *"Plan 01-03 Test
1's 'additive' contract is overridden per D-22"* — but the
override applies only to the `('helper', 'both')` branch. The
`'primary'` branch remains on the additive contract.

This is acceptable PER THE PLAN if the `'primary'` mode is
documented as non-production and the divergence is intentional.
The plan's comment block does not say that explicitly.

**Fix:**
Either (a) extend the B1 gate to the `'primary'` branch with an
identical `if not is_subcontractor_row:` clause (preserving the
plan's "primary mode is unchanged" rule by emitting a primary
key for non-sub rows and excluding sub rows — same shape as the
`'both'` branch); or (b) document the `'primary'` mode as
known-broken for subcontractor sheets via a startup banner
WARNING when `RES_GROUPING_MODE='primary' and SUBCONTRACTOR_RATE_VARIANTS_ENABLED`.

Option (b) is lower risk because the plan explicitly committed to
not touching the `'primary'` branch. Recommended text for the
WARNING:

```python
if (
    RES_GROUPING_MODE == 'primary'
    and SUBCONTRACTOR_RATE_VARIANTS_ENABLED

    logging.warning(
        "⚠️ RES_GROUPING_MODE=primary with SUBCONTRACTOR_RATE_VARIANTS_ENABLED "
        "produces DUPLICATE files for subcontractor WRs (legacy primary + "
        "_ReducedSub / _AEPBillable variants). Production runs in 'both' mode "

    )
```

### IN-01: Inner `from billing_audit.writer import lookup_attribution` inside the per-row loop

**File:** `generate_weekly_pdfs.py:4789`

**Issue:**
The reader import is hoisted INSIDE the per-row helper-shadow
construction block. The plan's stated rationale: *"kept local to
the call site to avoid polluting module imports for non-
subcontractor runs."*

Python's `import` is cached in `sys.modules`, so the second-and-later
imports are dict lookups — there is NO meaningful "pollution" cost
to a top-level import that already lives in the codebase. The
inner import does have one observable side effect: it forces a
fresh attribute lookup against `billing_audit.writer` each time the
block runs, which makes `mock.patch('billing_audit.writer.lookup_attribution')`
testing reliable.

A top-level import would NOT break the test (the patch replaces the
module attribute, and the test runs INSIDE
`with mock.patch(...):` so the patched attribute is fetched at line
4789 each iteration). The inner import works but is non-idiomatic.

**Fix:**
Hoist the import to the top of `generate_weekly_pdfs.py`:

```python
try:
    from billing_audit.writer import lookup_attribution
except ImportError:
    lookup_attribution = None  # billing_audit optional
```

Then the call site reads:

```python
if (
    is_subcontractor_row
    and SUBCONTRACTOR_HELPER_CLAIM_ATTRIBUTION_ENABLED
    and lookup_attribution is not None
):
    try:
        _attribution_row = lookup_attribution(...)
    except Exception:
        ...
```

This makes the import optional (matches the `BILLING_AUDIT_AVAILABLE`
pattern used elsewhere in this file for other billing_audit
symbols) and removes the per-row import overhead.

### IN-02: `from billing_audit.client import _global_disable_reason` imports a private symbol across package boundaries

**File:** `generate_weekly_pdfs.py:4828-4830`

**Issue:**
The reason-distinguishing logic at L4827-4836 reads:

```python
try:
    from billing_audit.client import (
        _global_disable_reason,
    )
    if _global_disable_reason is not None:
        _attribution_reason = 'fetch_failure'
    else:
        _attribution_reason = 'no_history'
except Exception:
    _attribution_reason = 'no_history'
```

`_global_disable_reason` is a private module-level variable
(leading underscore). Importing it from `generate_weekly_pdfs.py`
crosses the package boundary and creates a hidden coupling: any
rename or refactor in `billing_audit/client.py` silently breaks
this branch (the `except Exception` swallows the error and the
caller falls through to `no_history`, masking the regression).

The better contract is a public accessor in `billing_audit.client`:

```python
def is_globally_disabled() -> tuple[bool, str | None]:
    """Return (disabled, reason_code)."""
    return (_global_disable_reason is not None, _global_disable_reason)
```

…then the caller imports the function, not the variable.

The current pattern also exhibits the "import-once, observe-live"
property only because the `from ... import` statement runs INSIDE
the per-row block (re-importing each iteration). If a future
refactor hoists the import to the top of the file (per IN-01
above), the imported name would be a SNAPSHOT taken at module load
time — and the entire reason-distinguishing logic would silently
fail closed (always reporting `no_history`). This is a hidden
hazard for future refactoring.

**Fix:**
Add `is_globally_disabled()` (or similar) to
`billing_audit/client.py`'s public surface (no leading underscore,
documented in the docstring), then change the caller to:

```python
from billing_audit.client import is_globally_disabled
# ...
disabled, reason_code = is_globally_disabled()
if disabled:
    _attribution_reason = 'fetch_failure'
else:
    _attribution_reason = 'no_history'
```

### IN-03: `_attribution_row.get('helper')` — naming inconsistency with the dict's documented schema

`source_run_id` are intentionally ignored in v1.0 (and add a
deferred-ideas entry for using them in v1.1); or (b) extend the

```python
else:
```
