## Bug C — Attribution model (Q1: row partitioning shape)

| Option | Description | Selected |
|--------|-------------|----------|
| Per-row attribution | Each row appears ONLY in the helper file of whoever was the active 'Foreman Helping?' at the moment that row's 'Helping Foreman Completed Unit?' was first observed checked. Most accurate; drives toward attribution_snapshot / freeze_attribution. | ✓ |
| Date-cutoff partition | Single mid-week change-date acts as cutoff; rows before go to A, rows after go to B. Simpler; needs one event per WR per week. | |
| Status quo (no fix) | Both files contain full row set; defers Bug C to a later phase. | |
| Other / refine | Different model or alternative proposal. | |

**User's choice:** Per-row attribution.
**Notes:** Drives the source-of-truth toward `attribution_snapshot` (per-row freezes from `freeze_attribution`); needs data-team coordination for read access.

## Bug C — Attribution model (Q2: data source)

| Option | Description | Selected |
|--------|-------------|----------|
| Read from attribution_snapshot | Query the existing `billing_audit.attribution_snapshot` table. Needs data team to expose SELECT OR add a lookup RPC. Leverages existing data; one-time coordination cost. | ✓ |

## Bug C — Attribution model (Q3: failure mode)

| Option | Description | Selected |
|--------|-------------|----------|
| Fall back to current helper | Helper file includes ALL rows where current 'Foreman Helping?' matches this helper AND both checkboxes are checked (Phase 1 behavior). Logs a per-WR WARNING. Safe default. | ✓ |
| Emit empty helper file | Skip helper file emission until attribution is available. Logs a WARNING. Strictest fidelity. | |
| Hard fail the WR | Skip the entire WR group (no primary variants either) until attribution available. Most conservative; risks losing real billing data on transient outages. | |

**User's choice:** Fall back to current helper.
**Notes:** Helper files always emit (never silently empty); per-WR WARNING per failure mode (`disabled` / `no_history` / `fetch_failure`) gives operators visibility.

## Bug B2 — PPP cleanup defense

| Option | Description | Selected |
|--------|-------------|----------|
| Trust B1 + add cleanup whitelist | Both: (1) B1 fix self-resolves B2 on next run, AND (2) extend cleanup_untracked_sheet_attachments with per-sheet variant whitelist (PPP only accepts reduced_sub/reduced_sub_helper). Symmetric with round-9 target_map quarantine. | ✓ |
| Trust B1 fix alone | Only fix B1. Cleanup naturally recognizes stale file on next cron and deletes. Minimum scope; relies on B1 correctness. | |
| Cleanup whitelist only (skip B1 for B2) | B1 still fixed for its own sake, but B2 solved via cleanup-only. Conceptually wrong; not recommended. | |

**User's choice:** Trust B1 + add cleanup whitelist.
**Notes:** Defense-in-depth — routing AND cleanup contracts become symmetric per-sheet. Whitelist is hardcoded `{'reduced_sub', 'reduced_sub_helper'}` at the PPP call site; TARGET cleanup unchanged.

## Hash-history orphan handling

| Option | Description | Selected |
|--------|-------------|----------|
| Explicit one-time prune pass | Add `PHASE_1_1_HASH_PRUNE_VERSION` module constant; at session startup, if version advances, walk hash_history and drop subcontractor primary orphans. Idempotent. Loud INFO log. | ✓ |
| Let the 1000-entry cap evict naturally | Do nothing. Orphans sit until normal LRU/cap eviction. YAGNI; no extra code. Downside: temporary noise in hash_history.json. | |
| Operator RESET_WR_LIST | Document a manual operator step. Risks being forgotten; not idempotent. | |

**User's choice:** Explicit one-time prune pass.
**Notes:** Version constant IS the kill switch — advancing triggers the prune, leaving at current value skips it. No env var needed.
