# Phase 01.1 Plan 05: Closure (Hash Prune + E2E Tests + Living Ledger) Summary

Phase 1.1 closure plan landed. The five-plan / five-wave gap-closure
of three production bugs latent in Phase 01 (Bug A pre-acceptance
rescue gap, Bug B1 variant-tagging-additive, Bug B2 stale PPP
primary-shape file) plus one new feature (Bug C per-row
claim-history attribution) is now anchored against true end-to-end
test coverage. The structural weakness of
``TestHelperShadowVariantFileIdentifier`` (mirror class bypassing
the upstream classifier — the failure mode that let Phase 1 ship
with Bugs A and B1 latent in production) is closed by the new
``TestEndToEndPipeline`` + ``TestProductionCodeSiteInvariants``
classes in ``tests/test_subcontractor_helper_shadow_rescue.py``.

The SUB-12 idempotent hash-history one-time prune pass — drops
orphan subcontractor primary entries left behind by pre-Bug-B1
runs — lands gated by the ``PHASE_1_1_HASH_PRUNE_VERSION`` module
constant + persisted ``_phase_prune_version`` sentinel in
``hash_history.json``. The constant IS the kill switch (D-19);
advance to trigger; leave at the persisted value to skip.

The D-22 in-place rewrite of Plan 01-03 Test 1
(``test_kill_switch_disables_new_variant_emission``) extends the
test contract to assert that Bug B1 partitioning is permanent
(independent of the variant kill switch) — preserves
git-blame-traceability per RESEARCH.md §B1 by keeping the class
+ method name intact and adding a sibling
``test_partitioning_contract_for_subcontractor_non_helper_rows``
that asserts the post-Phase-1.1 invariant directly.

The D-23 Living Ledger entry permanently captures the four
design-rule lessons from Phase 1.1 in CLAUDE.md:
2-cycle ``/gsd-debug`` methodology, Plan 01-03 Test 1
design-intent override, pre-acceptance-rescue-generalization rule,
test-methodology rule.

## Key Decisions Made

- **Module-level helper extraction.** Extracted the prune-pass logic
  into `_run_phase_1_1_hash_prune(hash_history, groups) -> None` per
  the plan's optional Task 2 extraction note. The helper is callable
  directly by the regression suite via
  `TestHashPruneIdempotency`'s 6 test methods without spinning up
  `main()`. The session-startup site invokes the helper through a
  fail-safe `try/except` so a prune regression cannot break the
  billing pipeline.

- **Direct-helper test driver vs full `_fetch_and_process_sheet` mock.**
  Per RESEARCH.md Pitfall 6, the e2e test exercises the four
  upstream fixes through real production code but is authorized to
  drive each helper independently if full `_fetch_and_process_sheet`
  mocking proves heavy. Used the direct-driver pattern:
  `_subcontractor_rescue_price` for Bug A; `group_source_rows([...])`
  with post-rescue rows for Bug B1 / Bug C;
  `cleanup_untracked_sheet_attachments(..., variant_whitelist=...)`
  for Bug B2; `_run_phase_1_1_hash_prune(...)` for SUB-12. The
  source-grep guards in `TestProductionCodeSiteInvariants` close
  the gap by asserting the four production code-site invariants
  remain in place.

- **D-15 scope tightening on Bug C frozen-helper assertion.** The
  initial Bug C TestEndToEndPipeline assertion was too aggressive
  (asserted `ReplacementForeman` was absent from ALL emitted keys).
  D-15 explicitly scopes Bug C to subcontractor shadow-variant
  emission only — the legacy `_HELPER_<name>` group key continues
  to use the current Smartsheet `Foreman Helping?` value. The
  assertion was tightened to focus on the SHADOW-variant emission
  specifically (`REDUCEDSUB_HELPER_OriginalForeman` /
  `AEPBILLABLE_HELPER_OriginalForeman` MUST use the frozen helper;
  `_HELPER_ReplacementForeman` correctly persists the current
  helper).

- **APIError-driven test skip pattern.** TestLookupAttribution's
  4 APIError-driven test methods (transient retry exhaustion,
  permanent single-attempt, PGRST106 global kill, op-isolation
  invariant) use the existing `_POSTGREST_API_ERROR_CLS is None`
  `@unittest.skipIf` guard at the method level. This lets the 10
  non-APIError tests (RPC dispatch correctness, no-client, op
  identifier, WR sanitization, input validation, success path,
  helper-None / helper-empty-string / list-shape / empty-list)
  run unconditionally on every environment while skipping the 4
  APIError-driven tests cleanly when `postgrest` is not installed
  (mirrors `PostgrestErrorClassificationTests`).

- **D-22 rewrite preserves git-blame-traceability.** Per RESEARCH.md
  §B1, the Plan 01-03 Test 1 rewrite was in-place: the class name
  (`TestSubcontractorVariantKillSwitchAndScope`) and method name
  (`test_kill_switch_disables_new_variant_emission`) are
  UNCHANGED. The docstring was rewritten to cite the D-22
  design-intent override and the new Living Ledger entry. Existing
  assertions (kill switch suppresses `_AEPBILLABLE` / `_REDUCEDSUB`)
  PRESERVED. A NEW assertion was added that the legacy primary
  key is never emitted for subcontractor rows even with the kill
  switch OFF (Bug B1 partitioning is independent of the variant
  kill switch). A sibling
  `test_partitioning_contract_for_subcontractor_non_helper_rows`
  method was added in the same class asserting the post-Phase-1.1
  partitioning invariant directly.

- **Living Ledger timestamp set at commit time.** Per CONTEXT.md's
  "executor sets at commit time" discretion clause, the
  `[YYYY-MM-DD HH:MM]` placeholder was replaced with the real UTC
  timestamp at the moment of `git commit`: `[2026-05-20 00:26]`.
  The repo's current-date envelope says 2026-05-19, but `git
  commit` fired shortly after midnight UTC.

- **Markdown-lint-clean phrasing in the ledger entry.** The initial
  draft of the Living Ledger entry triggered three MD004
  (unordered-list-style) warnings on continuation lines that began
  with `  + ` at column 3 (the linter parses these as `+`-style
  bullets in a list that previously used `*`-style elsewhere).
  Rephrased the affected sentences to use "and" / "alongside" /
  "gain of N" so the entry parses as flat prose.

**1. [Rule 1 — Bug] Bug C TestEndToEndPipeline assertion over-constrained the legacy `_HELPER_<name>` flow**

- **Found during:** Task 2 (test creation), first pytest run reported
  `test_bug_c_attribution_partitions_row_to_frozen_helper` FAILED:
  the emitted group-keys list included
  `041926_91467680_HELPER_ReplacementForeman` (the legacy
  non-subcontractor helper file, which Bug C does NOT touch per
  D-15 scope).
- **Issue:** The initial `assertFalse(any('ReplacementForeman' in k
  for k in keys), ...)` asserted the current helper name was absent
  from ALL emitted keys — but D-15 explicitly preserves the legacy
  `_HELPER_<name>` flow's use of the current Smartsheet `Foreman
  Helping?` value. The over-constrained assertion would have
  caused PRs that correctly implement D-15 scope to fail this test.
- **Fix:** Tightened the assertion to focus on the shadow-variant
  emission specifically. The frozen helper MUST be in
  `REDUCEDSUB_HELPER_OriginalForeman` /
  `AEPBILLABLE_HELPER_OriginalForeman`; the current helper MUST
  NOT be in the shadow-variant slot
  (`REDUCEDSUB_HELPER_ReplacementForeman` /
  `AEPBILLABLE_HELPER_ReplacementForeman`). Added an explicit
  D-15 scope guard: the legacy `_HELPER_ReplacementForeman` key
  IS still expected to persist (with the current helper, not the
  frozen helper).
- **Files modified:** `tests/test_subcontractor_helper_shadow_rescue.py`
- **Commit:** `33a738f` (Task 2 commit — the fix landed in the same
  edit pass as the original test file)
