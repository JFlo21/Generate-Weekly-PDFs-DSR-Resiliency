# Phase 01.1 Plan 04: Bug C Claim-History Attribution Summary

Phase 1.1 Bug C (D-10..D-16 / SUB-11) closure landed. Subcontractor
helper-shadow rows are now partitioned by per-row claim history from
`billing_audit.attribution_snapshot` rather than the current Smartsheet
`Foreman Helping?` value. A foreman who helped Mon-Tue on a
subcontractor WR will keep their partitioned `_ReducedSub_Helper_<name>`
and `_AEPBillable_Helper_<name>` files even after a Wed swap on the
Smartsheet column — the production failure mode the 2-cycle debug
report (`.planning/debug/sub-helper-shadow-missing.md`) flagged as
Bug C.

The fix is four coordinated edits across five files:

1. **`billing_audit/writer.py`** — new `lookup_attribution(wr,
   week_ending, smartsheet_row_id) -> dict | None` reader function,
   structurally mirror-symmetric with `freeze_row`'s RPC dispatch.
   Uses a DISTINCT `op='lookup_attribution'` identifier so its
   per-op circuit breaker counter is independent of
   `freeze_attribution` / `pipeline_run_select` / `pipeline_run_upsert`
   (the op-isolation invariant from `[2026-04-25 12:00]`).
2. **`billing_audit/__init__.py`** — docstring updated to acknowledge
   the Phase 1.1 read-path arrival; `__all__` unchanged (existing
   `writer` re-export catches the new symbol).
3. **`generate_weekly_pdfs.py`** — module-level kill switch
   `SUBCONTRACTOR_HELPER_CLAIM_ATTRIBUTION_ENABLED` (default `'1'`),
   startup banner log, `_PII_LOG_MARKERS` entry, per-WR dedupe set
   at `group_source_rows` function scope, and the Bug C integration
   block INSIDE the existing helper-shadow construction block. The
   block invokes the reader ONLY when all four gates fire
   (`is_subcontractor_row AND kill-switch-on AND _valid_helper_row
   AND _helper_mode_enabled`), overrides `_attributed_helper` with
   the frozen helper name on success, and falls back to the current
   `helper_foreman` on `None` return (D-12 safe default).
4. **`.github/workflows/weekly-excel-generation.yml`** — replaces the
   Plan 01.1-01 placeholder comment with the actual
   `SUBCONTRACTOR_HELPER_CLAIM_ATTRIBUTION_ENABLED: '1'` pin plus a
   detailed rollback comment block.
5. **`website/docs/reference/environment.md`** — full operator-facing
   doc entry for the new env var (semantics, default, scope, fall-back
   contract, rollback path, workflow pin reference, data-team
   coordination note, startup banner output, per-WR WARNING body).

Phase 1 helper-shadow emission semantics are byte-identical for
non-subcontractor rows and for subcontractor rows when the reader
returns `None` (D-12 safe default) — ROADMAP success criterion #5
(byte-identical primary / helper / vac_crew / ORIG-folder outputs)
preserved.

## Key Decisions Made

- **Reader signature: positional `(wr, week_ending, smartsheet_row_id)`**
  to match the RPC's `p_wr` / `p_week_ending` / `p_smartsheet_row_id`
  parameter contract (RESEARCH.md §C). Test 1 in the plan asserts on
  this exact ordering.
- **No internal feature-flag check inside `lookup_attribution`** —
  the kill switch lives at the caller site (`group_source_rows`).
  Keeping the reader gate-free preserves testability and lets future
  callers reuse the reader for audit queries / debug tools without
  having to flip the production env var.
- **Empty-string helper from RPC treated as `None` at reader boundary**
  (`data.get('helper')` falsy short-circuit, Test 8 in the plan).
  PostgREST's NULL → empty-string convention would otherwise route
  rows to a `_REDUCEDSUB_HELPER_` file (the empty-string-helper
  variant), conflicting with the legacy primary file's naming
  convention.
- **Local import inside the integration block** (`from
  billing_audit.writer import lookup_attribution` at the call site)
  — keeps the module imports clean for non-subcontractor runs and
  tracks the Plan 01-04 pattern for per-block billing_audit imports.
- **Defensive `try/except` wrap around the reader call** —
  defense-in-depth beyond the reader's own `None`-return contract.
  Even if a future reader regression raises (e.g., an unexpected
  exception in the supabase client that escapes the reader's
  exception boundary), the pipeline MUST NEVER crash — the fall-back
  to current helper is the safe semantic.
- **Per-WR dedupe tuple key uses `week_end_for_key` (MMDDYY)** instead
  of the plan body's suggested `week_end_raw`. Reason: the
  `week_end_raw` variable is NOT actually in scope at the integration
  site inside `group_source_rows` — only `week_ending_date` (the
  parsed `date` object) and `week_end_for_key` (its MMDDYY-formatted
  string) are local. The existing Phase 1 helper diagnostic logs at
  L4337 use `week_end_for_key`; the dedupe set follows the same
  convention for consistency. Both are stable per-row so the dedupe
  semantics are unchanged.
- **`_attributed_helper` flows through the WHOLE shadow-variant
  emission block** — including the `keys_to_add.append(...)` tuple's
  third element (the foreman label) and the `Helper=<name>` log
  lines. Phase 1's emission body was passing `helper_foreman`
  directly; the Bug C change is precise: the `_attributed_helper`
  variable carries either the frozen helper (success path) or the
  current `helper_foreman` (D-12 fall-back), so substituting it
  everywhere preserves the byte-identical behavior on the fall-back
  path while activating per-row partitioning on the success path.
- **Bug C reader NOT parallelized in v1.0** — RESEARCH.md §C latency
  analysis (~12s wall-clock across ~100 helper-candidate rows ×
  ~120ms) is well within `TIME_BUDGET_MINUTES=95` (0.2% of budget).
  If a future cron run regresses, the parallelization pattern is
  documented in the plan `<deferred_ideas>` (`freeze_row`
  ThreadPoolExecutor template per `[2026-04-25 14:00]`).
- **Helper-set UNION D-13 INTENTIONALLY DEFERRED to a future phase**.
  Under the D-12 fall-back contract, every row that's still in the
  current Smartsheet snapshot is attributed correctly via its
  individual `lookup_attribution` call. The UNION (which requires a
  SECOND data-team RPC, `lookup_distinct_helpers_for_wr_week`) only
  matters when a helper attributed rows that have since been
  DELETED from Smartsheet — an edge case not present in the
  production failure modes Phase 1.1 targets.
- **Workflow placeholder REPLACED, not duplicated.** Plan 01.1-01's
  placeholder comment block at L345-346 was the explicit landing
  site for this plan's pin, per the wave-execution coordination
  documented in Plan 01.1-01's SUMMARY ("Plan 01.1-04 owns
  activating this pin"). The replacement removes the placeholder
  substring entirely so reviewers see exactly which feature state is
  active without having to grep for two strings.
