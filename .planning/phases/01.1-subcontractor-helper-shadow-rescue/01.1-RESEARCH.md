## Summary

Phase 1.1 is a four-bug hotfix bundle plus one new feature (claim-history attribution), all strictly scoped to subcontractor-folder sheets. The implementation is **almost entirely structural — every code path Phase 1.1 adds has a working analog already in the codebase**:

- **Bug A** mirrors the `[2026-04-23 00:00]` VAC-crew pre-acceptance rescue, structurally identical (one new `elif is_subcontractor_sheet and SUBCONTRACTOR_RATE_RECALC_PREACCEPTANCE_ENABLED:` branch alongside the existing primary-rate recalc block).
- **Bug B1** is a single boolean gate inversion at `generate_weekly_pdfs.py:4190` — the variable already exists, just needs hoisting.
- **Bug B2** is a per-sheet `variant_whitelist: set[str] | None = None` kwarg threaded through `cleanup_untracked_sheet_attachments`; the function already parses variants via `build_group_identity`.
- **Bug C** is the only path that introduces a genuinely new code surface — a `billing_audit.lookup_attribution` reader. The recommended PostgREST shape is a new RPC (cleanest dependency boundary; data team owns the function body).
- **Hash-history prune** is a one-time idempotent migration gated by a module constant + persisted sentinel.

**Primary recommendation:** Plan the work as **five tightly-scoped plans** in this dependency order: (1) Bug A rescue + kill switch + workflow pinning, (2) Bug B1 partitioning gate, (3) Bug B2 cleanup whitelist, (4) Bug C reader + main-loop integration, (5) hash-history prune + end-to-end integration test + Plan 01-03 Test 1 rewrite. Bug A and Bug B1 are independent of each other and can land in either order; Bug B2 self-resolves once B1 ships but the whitelist is belt-and-suspenders defense-in-depth (D-07 ships BOTH). Bug C and the hash-history prune are independent of A/B/B2 — they can be slotted anywhere. Living Ledger entry (D-23) ships in the final PR with the e2e test.

**Single critical research output — D-11 PostgREST access method:** Recommend a new `lookup_attribution(p_wr text, p_week_ending date, p_smartsheet_row_id bigint)` RPC that returns the frozen helper + source_run_id. Rationale: the `attribution_snapshot` table schema lives entirely in the data team's Supabase Dashboard (NOT in `billing_audit/schema.sql`), so the cleanest dependency boundary is another RPC — same pattern as the existing `freeze_attribution` RPC. See "Architecture Patterns" §C for the call shape, error classification, and per-row latency budget analysis.

### Alternatives Considered

| Instead of | Could Use | Tradeoff |
|------------|-----------|----------|
| `lookup_attribution` RPC (recommended) | Direct `.schema('billing_audit').table('attribution_snapshot').select(...)` | Direct SELECT requires the data team to add `billing_audit.attribution_snapshot` to the PostgREST "Exposed schemas" list. This is one extra Dashboard step but already required by the existing `pipeline_run` SELECT (which works), so the schema IS already exposed. Tradeoff: a direct SELECT couples the Python code tightly to `attribution_snapshot`'s exact column shape (column-rename in Supabase silently breaks the reader). An RPC abstracts the contract behind a named function — same pattern as `freeze_attribution`. **Researcher recommendation: prefer the RPC for symmetry with `freeze_attribution`, but if the data team prefers direct SELECT, that is also viable** because the schema is already exposed. |
| Caching `lookup_attribution` results across all rows in a WR group | Per-row reads (recommended) | A single-call-per-group reader returning `dict[row_id, helper_foreman]` would amortize HTTP latency across all rows. However, the data team would need to add a `lookup_attribution_for_wr_week(p_wr, p_week_ending) RETURNS SETOF ...` RPC. This is a Phase 1.2 optimization — defer per CONTEXT.md "Deferred Ideas" (parallelize Bug C reader). |
| Storing the frozen helper on `pipeline_run.variant` | Reading from `attribution_snapshot` (per D-11) | Rejected by D-16: "No further schema changes required for Bug C." The `pipeline_run.variant` column is a categorical token, not a personnel identifier. `attribution_snapshot` is the correct table — it already carries `p_helper` per the `freeze_attribution` RPC's `p_helper TEXT` parameter contract documented in `billing_audit/schema.sql` L141. |

### Pattern §A1 — Bug A pre-acceptance rescue (mirror `[2026-04-23 00:00]` VAC-crew fix)

**What:** Extend the rate-recalc block at `generate_weekly_pdfs.py:3576-3651` with a second branch for subcontractor sheets. The new branch mutates `price_val` in-place using `_SUBCONTRACTOR_RATES` for the row's CU code + work type. Helper-detection at L3683-3714 then fires as designed.

**When to use:** Triggered when ALL of the following are true:
1. `is_subcontractor_sheet` is True (existing per-sheet flag in `_fetch_and_process_sheet`)
2. `SUBCONTRACTOR_RATE_RECALC_PREACCEPTANCE_ENABLED` env var is truthy
3. The row's `Units Total Price` is blank/zero (so `has_price` would otherwise be False)
4. `_SUBCONTRACTOR_RATES` carries a rate entry for the row's CU code (Bug A only fires when a rate is FOUND; missing CU falls through to SmartSheet safety floor)

**Researcher recommendation — rescue helper shape (Discretion D-01):** **Use a stripped local helper, not a full `_resolve_row_price` second pass.** Rationale:

### Pattern §HP — Hash-history one-time prune

**What:** At session startup, immediately after `hash_history = load_hash_history(HASH_HISTORY_PATH)`, check `hash_history.get('_phase_prune_version', 0)`. ### Anti-Patterns to Avoid

- **Mixing primary recalc gate with subcontractor rescue gate.** Bug A's new branch MUST be a SEPARATE `elif is_subcontractor_sheet and SUBCONTRACTOR_RATE_RECALC_PREACCEPTANCE_ENABLED:` block — NOT a modification to the existing `if RATE_CUTOFF_DATE and _rate_new_primary and not is_subcontractor_sheet:` branch. The two pricing systems must not be tangled.
- **Mutating `_resolve_row_price` to add a "rescue mode."** Per Discretion D-01: minimum-diff helper is preferred over refactoring an existing variant-aware function.
- **Broadening the cleanup whitelist to include `'helper'` or `'vac_crew'` on PPP.** D-09: PPP only receives `reduced_sub` and `reduced_sub_helper` from Phase 1's routing matrix. Any other variant on PPP is by definition a stale upload from a prior period and must be deleted.
- **Caching `lookup_attribution` results across runs.** The reader's result is per-(wr, week, row_id). Phase 1.1 has no cross-run cache for this — every run re-queries (the per-row cost is bounded; see latency budget above). If a future phase introduces caching, the cache MUST be invalidated when the row's `Helping Foreman Completed Unit?` is unchecked (the freeze record then becomes stale-but-still-referenced).
- **Using the legacy `_HELPER_<name>` group_key shape for subcontractor helper files.** Phase 1 already created `_REDUCEDSUB_HELPER_<name>` and `_AEPBILLABLE_HELPER_<name>` shapes. Bug C must produce files keyed under THESE existing shadow variants, NOT under a new variant string. The variant set stays at 7 (no SUB-13 / SUB-14 introduced by Phase 1.1).
- **Allowing Bug A's rescue to apply to non-subcontractor sheets.** The gate condition (`is_subcontractor_sheet`) MUST be the first condition in the rescue branch. The existing primary-rate recalc at L3576-3651 is the rescue path for non-subcontractor sheets — these are NOT to be merged.
- **Re-introducing the additive contract in Plan 01-03's test (D-22).** The test rewrite must explicitly assert the absence of the legacy primary group key for subcontractor rows.

## Common Pitfalls

### Pitfall 1: Mutating `_resolve_row_price` to add a rescue mode

**What goes wrong:** Phase 1's `_resolve_row_price` is variant-aware. A naive Bug A implementation passes `variant='reduced_sub'` to `_resolve_row_price` at the pre-acceptance gate — but the row hasn't been classified as a variant yet at that point. The classification happens in `group_source_rows` AFTER row acceptance. The coupling creates a circular dependency.

**Why it happens:** `_resolve_row_price` looks like the "right" helper because it's variant-aware. The temptation is to reuse it. But it's the WRONG abstraction for pre-acceptance rescue — at that point we're not pricing a row for output, we're rescuing it for acceptance.

**How to avoid:** Use a stripped local helper (`_subcontractor_rescue_price`) that takes only the row dict and returns a rescue price OR 0.0. Never variant-aware.

**Warning signs:** A reviewer asks "why are we computing variant before the row is grouped?" or "what should the variant be if Snapshot Date isn't yet known?"

### Pitfall 2: PostgREST schema-exposure drift

**What goes wrong:** The data team's Supabase project's "Exposed schemas" list may not include `billing_audit` after a Supabase upgrade or accidental config change. The reader returns PGRST106 on every call.

**Why it happens:** Supabase Dashboard config drift; key rotation.

**How to avoid:** The existing `_classify_postgrest_error` already handles PGRST106 as a run-global kill — `get_client()` returns None for the rest of the run, every Bug C reader call fast-fails to D-12's `fetch_failure` fallback. **The pipeline is safe by construction.**

**Warning signs:** A scheduled run logs `🔌 billing_audit disabled for this run (code=PGRST106)`. Operators have a documented runbook (`[2026-04-24 10:50]` ledger entry) — same hint, same fix.

### Pitfall 3: Helper SET enumeration missing ever-frozen helpers

**What goes wrong:** A foreman helped Monday-Tuesday, then left the WR. The current Smartsheet snapshot shows a different helper as the active `Foreman Helping?`. Phase 1's current code emits a helper file ONLY for the active helper — the Mon-Tue helper's rows are still in the file (Bug C unfixed) OR are completely missing (Bug C fixed naively).

**Why it happens:** Naive Bug C implementations only query attribution_snapshot for the helpers present on the current Smartsheet snapshot. Helpers who left are invisible to that query.

**How to avoid:** Implement the UNION per D-13. Two operations: (1) current helpers from `group_source_rows`'s in-loop set; (2) ever-frozen helpers from a `SELECT DISTINCT helper FROM attribution_snapshot WHERE wr = ? AND week_ending = ?` query.

**Warning signs:** Operators report "Helper file for foreman X is missing on a WR where X helped earlier in the week." Test catch: synthetic fixture with `freeze_row` history for a helper who is no longer in the current Smartsheet snapshot.

### Pitfall 4: Hash-history prune key-format brittleness

**What goes wrong:** The `_phase_prune_version` sentinel key is filtered out by the existing `load_hash_history`'s validation `isinstance(v, dict) and 'hash' in v`. The sentinel is dropped on every load. The prune fires on every run (not idempotent).

**Why it happens:** `load_hash_history` was designed to drop malformed entries; an integer-valued sentinel key looks malformed to it.

**How to avoid:** Extend the `load_hash_history` filter to preserve sentinel keys (`startswith('_')`). Add a regression test that asserts `_phase_prune_version` survives a round-trip through `save_hash_history` → `load_hash_history`.

**Warning signs:** Logs show `🧹 Phase 1.1 hash-history prune (version 0 → 1): dropped N subcontractor primary orphan(s)` on every run instead of just the first. Test catch: `tests/test_subcontractor_helper_shadow_rescue.py::test_hash_prune_idempotency`.

### Pitfall 5: `is_subcontractor_row` hoist breaks an existing scope assumption

**What goes wrong:** Phase 1 added the `is_subcontractor_row` computation at L4244-4248 specifically AFTER the legacy primary cascade (L4181-4214). The order was deliberate — the legacy cascade depends on `is_vac_crew_row` (computed earlier) and `valid_helper_row` (computed earlier) but NOT on `is_subcontractor_row`. Hoisting `is_subcontractor_row` UP is safe IF the dependencies (`r.get('__source_sheet_id')` + `_FOLDER_DISCOVERED_SUB_IDS`) are both in scope at the hoist point. Both ARE in scope at function entry, so the hoist is safe — but a reviewer may flag this as a potential ordering risk.

**Why it happens:** Function ordering carries implicit assumptions about variable scope.

**How to avoid:** When hoisting, explicitly cite the verified dependencies in a comment. The end-to-end test (D-20) catches any actual breakage.

**Warning signs:** Mysterious `NameError: name '_row_sheet_id' is not defined` in CI — would only happen if the hoist site is OUTSIDE the per-row for-loop. Plan must keep the hoist INSIDE the for-loop body.

### Pitfall 6: `TestHelperShadowVariantFileIdentifier` is a structural mirror, not an end-to-end test

**What goes wrong:** The test class at `tests/test_subcontractor_pricing.py:3053-3252` builds its own helper `_derive_main_loop_identifier` that mirrors the production code. The test passes EVEN IF the production code's upstream classifier never fires on subcontractor rows. Plan 01 shipped with passing tests AND with Bugs A and B1 latent in production.

**Why it happens:** Mirror tests are easy to write but bypass the full pipeline. They only catch identifier-derivation regressions, not row-flow regressions.

**How to avoid:** D-20 mandates a TRUE end-to-end integration test driving `_fetch_and_process_sheet` → `group_source_rows` → `generate_excel`. Use `unittest.mock.MagicMock` to fake the Smartsheet `client.Sheets.get_sheet(...)` response with synthetic helper-row data. **Critical: feed rows whose `Units Total Price` is BLANK (Bug A precondition) AND whose `__source_sheet_id` is in `_FOLDER_DISCOVERED_SUB_IDS` (Bug B1 + Bug C precondition).** Test assertions:
- (a) After the rescue + grouping, the `groups` dict contains `_REDUCEDSUB_HELPER_<foreman>` keys and NO bare-primary keys.
- (b) After `generate_excel`, the on-disk filenames include `_ReducedSub_Helper_<foreman>` and NOT a bare-primary filename.
- (c) When `SUBCONTRACTOR_RATE_RECALC_PREACCEPTANCE_ENABLED='0'`, the helper rows drop at the price gate (Bug A regression guard).
- (d) When `SUBCONTRACTOR_HELPER_CLAIM_ATTRIBUTION_ENABLED='1'` and the mock `lookup_attribution` returns a frozen helper differing from the current Smartsheet `Foreman Helping?`, the row appears in the frozen helper's file (Bug C invariant).

**Warning signs:** A reviewer asks "what does this test actually exercise end-to-end?" — answer: the synthetic payload travels through the full pipeline, and the test asserts on file paths and group keys, NOT on internal helper return values.

### Pitfall 7: Living Ledger entry omits the test-methodology rule (D-23 rule d)

**What goes wrong:** A short Living Ledger entry skips the "any plan that fixes a row-flow bug MUST add at least one true end-to-end test driving the full pipeline" rule. The next post-Phase-1.1 row-flow bug repeats the Plan 01 test-coverage gap.

**Why it happens:** Ledger entries are easy to under-document under time pressure.

**How to avoid:** D-23 explicitly enumerates four rules. Plan must lock the entry's outline to all four numbered items.

**Warning signs:** Future debug session for a row-flow bug finds `TestHelperShadowVariantFileIdentifier`-style mirror tests passing while production is broken. The Phase 1.1 Living Ledger entry's rule (d) becomes the citation.

### §2 Subcontractor rescue helper (stripped variant of `_resolve_row_price`)

```python
# Source: generate_weekly_pdfs.py:1409-1541 (_resolve_row_price — canonical
# keys + work-type matcher + safety floor pattern)
# Source: PR #206 (`74cd2aa`) — abbreviated work-type matcher

def _subcontractor_rescue_price(row_data: dict) -> float:
    """Phase 1.1 Bug A — pre-acceptance rescue. Returns reduced-sub
    price × qty OR 0.0 (caller observes no rescue and the row drops
    at the existing has_price gate, same as legacy behaviour). Uses
    reduced_*_price as the safety floor — AEP cutoff is irrelevant
    here because we're rescuing for ACCEPTANCE, not pricing for
    output.
    """
    cu = str(row_data.get('CU') or '').strip().upper()
    rate_row = _SUBCONTRACTOR_RATES.get(cu)
    if rate_row is None:
        return 0.0
    work_type_raw = (row_data.get('Work Type') or '').strip().lower()
    if 'inst' in work_type_raw:
        rate = rate_row.get('reduced_install_price', 0.0)
    elif 'rem' in work_type_raw:
        rate = rate_row.get('reduced_remove_price', 0.0)
    elif 'tran' in work_type_raw or 'xfr' in work_type_raw:
        rate = rate_row.get('reduced_transfer_price', 0.0)
    else:
        return 0.0
    qty_raw = row_data.get('Quantity', 0)
    try:
        qty = float(qty_raw) if qty_raw not in (None, '') else 0.0
    except (TypeError, ValueError):
        qty = 0.0
    if rate <= 0 or qty <= 0:
        return 0.0
    return rate * qty
```

## Open Questions

1. **Should `lookup_attribution` reuse `pipeline_run_select`'s op identifier or get its own?**
   - What we know: `_classify_postgrest_error` is shared; the per-op circuit breaker (`_consecutive_failures` / `_open_circuits`) is per-op.
   - What's unclear: Whether a `lookup_attribution` outage should trip a breaker independent of `pipeline_run_select` or share it.
   - Recommendation: **Independent op** (`op="lookup_attribution"`). Rationale: an outage on the attribution lookup must not cascade into disabling the pipeline_run upsert (which is correctness-critical for drift detection). The existing per-op breaker contract from `[2026-04-25 14:00]` already guarantees this isolation.

2. **Should the end-to-end test live in `tests/test_subcontractor_pricing.py` or a NEW `tests/test_subcontractor_helper_shadow_rescue.py`?**
   - What we know: D-20 mandates an e2e test; existing `TestHelperShadowVariantFileIdentifier` is structurally weak and not deleted.
   - What's unclear: Whether to extend the existing file or split.
   - Recommendation: **New file `tests/test_subcontractor_helper_shadow_rescue.py`** to separate end-to-end pipeline tests from per-unit tests. Existing helpers (`_ensure_smartsheet_mocked`, `_reset_all`) can be moved to a shared `conftest.py` or imported.

3. **Should the Bug A rescue stamp a marker field on the row dict for downstream telemetry?**
   - What we know: The rescue mutates `price_val` in-place at the `_fetch_and_process_sheet` scope; the downstream `_resolve_row_price` at `generate_excel` does NOT need the marker (it re-computes from `_SUBCONTRACTOR_RATES`).
   - What's unclear: Whether operators want a `__subcontractor_rescued = True` flag for diagnostic logs / Sentry breadcrumbs.
   - Recommendation: **Add the flag** as a defensive telemetry hook — costs ~2 lines, supports the post-merge production-cron verification step.

4. **What happens to `_phase_prune_version` on `RESET_HASH_HISTORY=true`?**
   - What we know: `RESET_HASH_HISTORY=true` returns `{}` from `load_hash_history` (existing behaviour at L2720-2722).
   - What's unclear: After reset, the prune block sees version=0 and runs. There's nothing to prune (empty dict) but the sentinel gets written back. This is correct — `save_hash_history` will persist `_phase_prune_version=1`.
   - Recommendation: **No special-case needed.** The empty-dict path is observationally equivalent to "no orphans exist" — the prune block logs "no orphans to drop" and persists the sentinel. Regression test should cover this case.
