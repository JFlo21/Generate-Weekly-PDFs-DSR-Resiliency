# Phase 1.1: Subcontractor Helper-Shadow Rescue + Variant Partition + Claim-History Attribution - Context

**Gathered:** 2026-05-19
**Status:** Ready for planning

<domain>
## Phase Boundary

Urgent hotfix bundle for three production bugs discovered after Phase 1
(PR #203 + #206) shipped, plus one new feature (claim-history
attribution for subcontractor helper line items). Strictly scoped to
subcontractor-folder sheets (`SUBCONTRACTOR_FOLDER_IDS`-discovered).
Primary, helper, vac_crew, and original-contract-folder pipelines must
remain byte-identical (success criterion #5).

**Four problems addressed:**

1. **Bug A — Pre-acceptance helper-row rescue gap (Phase 1 oversight).**
   Subcontractor helper rows drop at the row-acceptance gate at
   `generate_weekly_pdfs.py:3669` because `has_price=False` (operator
   workflow leaves `Units Total Price` blank/zero while helper work is
   awaiting acceptance). Phase 1's `_resolve_row_price` lives downstream
   inside `generate_excel` (~L4839) and cannot rescue these rows. The
   helper-detection block at L3683-3714 lives INSIDE the same `if` arm,
   so `__is_helper_row` is never set and no shadow group key is
   instantiated. Result: zero `_ReducedSub_Helper_<name>` / 
   `_AEPBillable_Helper_<name>` files in production.

2. **Bug B1 — Variant tagging is additive, not partitioning, for
   subcontractor rows.** `group_source_rows` at L4181-4350 appends
   THREE keys to `keys_to_add` for every non-helper subcontractor row:
   legacy `primary`, `reduced_sub`, and (when post-cutoff)
   `aep_billable`. The legacy primary file lands on TARGET_SHEET_ID
   with content byte-equivalent to the `_ReducedSub` file
   (SmartSheet pricing on sub sheets is operator-configured to match
   reduced-sub CSV rates). Overrides the Plan 01-03 Test 1 "additive"
   design contract for subcontractor rows only — primary /
   original-contract / vac_crew rows continue to emit the legacy
   primary key unchanged.

3. **Bug B2 — Stale primary-shape file on SUBCONTRACTOR_PPP_SHEET_ID.**
   A historical attachment from a pre-Phase-01-routing-matrix period
   sits on PPP. The Phase 1 cleanup at L7196 cannot remove it because
   `valid_wr_weeks` is shared across both target sheets and Bug B1
   adds `(wr, week, 'primary', '')` to that set every run,
   legitimizing the stale file. Self-resolves once B1 lands; this
   phase adds a per-sheet variant whitelist as belt-and-suspenders.

4. **Bug C — Claim-history attribution for subcontractor helper files
   (NEW feature).** Helper files for a subcontractor WR currently
   contain the full row set regardless of which helper actually
   claimed each row. Phase 1.1 partitions helper file row sets by
   per-row attribution: each row appears only in the helper file of
   whoever was the active `Foreman Helping?` when that row's
   `Helping Foreman Completed Unit?` was first observed checked.
   Source of truth: existing `billing_audit.attribution_snapshot`
   (populated by the `freeze_attribution` RPC). Subcontractor
   workflow only — primary workflow's helper logic stays untouched.

## Implementation Decisions

### Bug A — Pre-acceptance helper-row rescue (locked by ROADMAP success criterion #1, #6)

- **D-01:** Extend the pre-acceptance rate-recalc rescue (the symmetric
  fix from `[2026-04-23 00:00]` Living Ledger entry for VAC-crew) to
  subcontractor sheets. The rescue invokes subcontractor-rate pricing
  BEFORE `has_price` is evaluated at L3669, using `_SUBCONTRACTOR_RATES`
  for the row's CU code + work type. Exact rescue shape (full
  `_resolve_row_price` call vs stripped helper) is a planner decision —
  preserve the structural pattern of the existing primary-rate
  Weekly-Ref-Date fallback at L3576-3651.
- **D-02:** Kill switch: `SUBCONTRACTOR_RATE_RECALC_PREACCEPTANCE_ENABLED`,
  default `'1'` (truthy values `1`/`true`/`yes`/`on`). Locked by
  ROADMAP success criterion #6. Default-ON per the `[2026-04-23 00:00]`
  rule that any pre-acceptance / data-shape change must ship with a
  rollback flag. Setting `'0'` reverts Bug A behavior to the pre-fix
  state without affecting Bug B1, B2, or Bug C. Pinned in the workflow
  `env:` block per IN-04.
- **D-03:** Rescue rate sourcing reuses the existing
  `_SUBCONTRACTOR_RATES` dict loaded once at session startup
  (Phase 1 plan 01-01). The rescue path MUST NOT re-read the CSV; it
  reads the cached dict. The same CU-lookup-then-fallback-to-SmartSheet
  pricing safety floor (D-16 from Phase 01 CONTEXT.md) applies — when
  no rate is found for a CU, the row falls through to SmartSheet
  pricing unchanged (no rescue, no error).

### Bug B1 — Variant partitioning for subcontractor rows (locked by ROADMAP success criterion #2)

- **D-04:** In `group_source_rows`, gate the legacy primary emission
  at L4185-4189 on `not is_subcontractor_row` for the non-helper
  branch. For subcontractor non-helper rows, emit ONLY the variant
  keys (`_REDUCEDSUB`, plus `_AEPBILLABLE` when post-cutoff). The
  helper subcontractor branch is already correctly partitioned via
  L4187's `if not valid_helper_row:` (helper rows already skip the
  primary key). Variable scope: `is_subcontractor_row` is currently
  computed AFTER the legacy primary emission at L4242; planner must
  either hoist the computation, gate at a later site, or recompute
  in-place. Reuse the existing `_FOLDER_DISCOVERED_SUB_IDS`
  membership check exactly as the existing site does — no new
  helpers.
- **D-05:** No kill switch for B1. Pure correctness fix per the
  standard "correctness fixes don't get flags" rule. Without the
  flag, B1 is permanent on merge.
- **D-06:** Primary / original-contract / vac_crew rows continue to
  emit the legacy primary key unchanged. The `is_subcontractor_row`
  gate scopes the change strictly to subcontractor rows — other
  sheet types fall through with zero behavior change. Byte-identical
  guarantee from ROADMAP success criterion #5 stays intact.

### Bug B2 — PPP cleanup defense (locked by ROADMAP success criterion #1, #3 orphan elimination)

- **D-07:** Both fixes ship: (1) trust B1's structural self-resolution
  AND (2) add per-sheet variant whitelist to
  `cleanup_untracked_sheet_attachments`. The whitelist is the inverse
  of the round-9 target_map collision quarantine rule
  (`decisions.md` 2026-04-23 21:00) — routing AND cleanup contracts
  become symmetric per-sheet. SUBCONTRACTOR_PPP_SHEET_ID accepts only
  `reduced_sub` and `reduced_sub_helper` variants; any other variant
  parsed from an attachment filename on PPP is unconditionally treated
  as off-contract and deleted (NOT just "not in `valid_wr_weeks`" —
  variant-set-membership is checked first).
- **D-08:** No kill switch for the whitelist. Same reasoning as
  Bug B1 — defense-in-depth correctness, very narrow scope (PPP
  only). The whitelist set is hardcoded (`{'reduced_sub',
  'reduced_sub_helper'}`) in the cleanup call site — no env var,
  no config.
- **D-09:** TARGET_SHEET_ID cleanup contract is unchanged. TARGET
  accepts every variant currently routed to it (primary, helper,
  vac_crew, aep_billable, reduced_sub, aep_billable_helper,
  reduced_sub_helper). The whitelist is per-sheet — passed as an
  argument to `cleanup_untracked_sheet_attachments` only when called
  for PPP. Planner must thread the whitelist through the existing
  cleanup invocation sites without changing TARGET behavior.

### Bug C — Per-row claim-history attribution (locked by ROADMAP success criterion #3)

- **D-10:** Helper row partitioning is PER-ROW, not date-cutoff. Each
  row appears ONLY in the helper file of whoever was the active
  `Foreman Helping?` at the moment that row's `Helping Foreman
  Completed Unit?` was first observed checked. Rejects the simpler
  "single date-cutoff partition" alternative because real-world
  helper switches frequently overlap with selective row completion
  (Foreman A completes rows 1-5 Mon-Tue; Foreman B replaces A and
  completes rows 6-10 Wed-Thu, but rows 11-15 stay completed by A
  on Friday under A's continued primary role).
- **D-11:** Source of truth: `billing_audit.attribution_snapshot`
  (already populated row-level by every cron run via the
  `freeze_attribution` RPC; first-write-wins semantics already match
  the per-row "first observed checked" definition). Reader queries
  for `(p_wr, p_week_ending, p_smartsheet_row_id)` and returns the
  frozen helper foreman (and `source_run_id` for audit). Does NOT
  add a new table; does NOT add a new column to `pipeline_run`. Does
  NOT issue Smartsheet cell-history calls (would blow the API
  budget). **Operator coordination required:** the data team must
  expose `attribution_snapshot` for SELECT in PostgREST's exposed
  schemas OR add a `lookup_attribution(p_wr, p_week_ending,
  p_row_id)` RPC. Schema discovery + access method is a RESEARCH
  task for the researcher to confirm before plan-phase.
- **D-12:** Fall-back-to-current-helper failure mode. When the
  attribution_snapshot read fails OR returns no row (first cron run
  for a WR, billing_audit disabled, transient Supabase outage),
  the helper file falls back to Phase 1 behavior: includes ALL rows
  where the current `Foreman Helping?` matches this helper's name
  AND both helper checkboxes are checked. Logs ONE per-WR WARNING
  naming the reason (`disabled` / `no_history` / `fetch_failure`).
  Safe default: helper files always emit (never silently empty),
  operators see the WARNING and can investigate.
- **D-13:** Helper file SET enumeration: the set of helper foremen
  for whom we generate `_ReducedSub_Helper_<name>` and
  `_AEPBillable_Helper_<name>` files for a given (WR, week) is the
  UNION of (a) currently-active helpers on the Smartsheet snapshot
  AND (b) every distinct ever-frozen helper for that (WR, week) in
  `attribution_snapshot`. A foreman who helped Mon-Tue, left, and is
  no longer the active `Foreman Helping?` still receives their
  partitioned helper file containing the rows they were attributed.
  This is the whole point of Bug C — file emission must outlive the
  Smartsheet column's current value.
- **D-14:** Kill switch: `SUBCONTRACTOR_HELPER_CLAIM_ATTRIBUTION_ENABLED`,
  default `'1'` (truthy values `1`/`true`/`yes`/`on`). Setting `'0'`
  reverts to Phase 1's full-row-set helper behavior (the same path as
  D-12 fall-back-to-current-helper, just unconditionally). Pinned in
  the workflow `env:` block per IN-04. Per the `[2026-04-23 00:00]`
  rule: any new behavioral data-shape change ships with a flag.
- **D-15:** Subcontractor workflow ONLY. The primary workflow's
  helper file behavior (legacy `_Helper_<name>` on primary sheets)
  is NOT touched. ROADMAP "don't damage" constraint plus byte-
  identical guarantee (#5). Planner must ensure the new attribution
  read is gated on `is_subcontractor_row` (or equivalent variant
  classification) at every site.
- **D-16:** `pipeline_run.variant` column (Phase 01 plan 05) is
  already in place and carries `reduced_sub_helper` /
  `aep_billable_helper` values. No further schema changes required
  for Bug C beyond what's already deployed. The new read code adds
  query surface but no write surface in pipeline_run.

### Hash-history orphan handling (locked by ROADMAP success criterion #5 byte-identical guarantee)

- **D-17:** Explicit one-time prune pass. Add a module-level constant
  `PHASE_1_1_HASH_PRUNE_VERSION = 1` (or similar). At session
  startup, after `hash_history` is loaded, if the version has
  advanced since the last persisted version (tracked in
  `hash_history.json` under a `_phase_prune_version` key), walk
  `hash_history` and drop entries whose group_key parses to
  (subcontractor WR + `primary` variant + empty identifier). Log
  ONE INFO line naming the count and the affected WR list. Persist
  the new version into `hash_history.json` so the prune is
  idempotent across runs.
- **D-18:** Affected-WR detection: an entry is "subcontractor
  primary orphan" if its parsed (wr, week, variant, identifier)
  tuple has `variant == 'primary'` AND `identifier == ''` AND
  membership in `_FOLDER_DISCOVERED_SUB_IDS` resolves true for at
  least one source sheet for that WR in THIS run's discovery
  result. The planner may simplify to "all primary-variant
  entries for WRs that produce reduced_sub keys this run" if the
  per-WR source-sheet membership map is awkward to thread.
- **D-19:** No kill switch for the prune pass. Idempotent + bounded
  in scope (only subcontractor primary orphans, only on version
  advance). The version constant IS the kill switch — leave it at
  the deployed value to skip the prune, advance to trigger it.

### Test methodology (locked by ROADMAP success criterion #4)

- **D-20:** True end-to-end integration test required. Drives
  `_fetch_and_process_sheet` → `group_source_rows` →
  `generate_excel` on a synthetic Smartsheet row payload with
  helper data. Closes the structural weakness of the existing
  `TestHelperShadowVariantFileIdentifier` (which bypasses the
  upstream classifier and the `has_price` gate and so passes
  regardless of whether Bug A or Bug B1 occurs). Fixture shape +
  file location is a planner decision; the test must exercise the
  full row → group → file pipeline.
- **D-21:** Regression tests per success criterion #4: (a)
  pre-acceptance rate rescue (Bug A — helper row with blank
  SmartSheet price + populated subcontractor rate ≠ dropped at
  L3669; reaches helper detection; produces shadow group key);
  (b) variant-partition gate (Bug B1 — non-helper subcontractor
  row produces only `_REDUCEDSUB` and `_AEPBILLABLE` keys, never
  legacy primary; rewrites Plan 01-03 Test 1 expectation);
  (c) PPP cleanup whitelist (Bug B2 — primary-shape attachment on
  PPP is deleted regardless of `valid_wr_weeks` state);
  (d) claim-history attribution (Bug C — row appears only in
  attribution-snapshot's frozen helper's file; current-helper
  fallback fires when read fails); (e) hash-history prune
  idempotency (Phase 1.1 version advances → orphan entries drop on
  first run; subsequent runs are no-op).
- **D-22:** Plan 01-03 Test 1 rewrite required. The existing test
  asserts the additive contract ("produces group keys
  `_AEPBILLABLE` AND `_REDUCEDSUB` in addition to the existing
  primary group key"). After Bug B1 lands, the assertion becomes
  "produces ONLY `_AEPBILLABLE` AND `_REDUCEDSUB`; no primary key
  for subcontractor rows". The planner must update Plan 01-03's
  test (or supersede it with a new test in a Phase 1.1 plan) and
  add a Living Ledger entry documenting the design-intent
  override per `decisions.md` 2026-05-15 12:00 rule on cross-plan
  consistency.

## Specific Ideas

- **Bug C "fall back to current helper" failure mode includes the
  no-history case explicitly.** First cron run for a brand-new WR
  (zero rows in attribution_snapshot) is not an error condition —
  it's the expected first state. The WARNING reason in that case
  is `no_history`, not `fetch_failure`; the per-WR WARNING fires
  ONCE per (WR, week, helper) tuple, not per-row, so the log
  doesn't drown.

- **"Currently-active helper" enumeration uses the Smartsheet
  snapshot, not the attribution_snapshot table.** D-13 unions
  (a) helpers present on the Smartsheet `Foreman Helping?` column
  on rows with both helper checkboxes checked AND (b) helpers
  frozen in attribution_snapshot for this (WR, week). Both sources
  contribute to the helper file SET; per-row partitioning (D-10)
  then assigns each row to ONE of those helpers' files.

- **Hash-history prune is opt-in via version constant, not env
  var.** Per D-19, `PHASE_1_1_HASH_PRUNE_VERSION = 1` (module
  constant) IS the kill switch. Leaving it at 0 (or whatever is
  already persisted in `hash_history.json`) skips the prune.
  Operators MUST NOT need to set an env var to opt in/out — the
  version advance is a code-controlled, code-reviewable trigger.

- **Plan 01-03 Test 1's "additive" assertion is the design-intent
  override boundary.** The Living Ledger entry (D-23 rule b) must
  cite Plan 01-03 Test 1 by name and explain that the design
  contract was correct for the time it was written but is now
  superseded by operator-product expectation (Bug B clarification
  2026-05-19). Future Phase 1 review work must reference both
  plans in sequence.

- **The byte-identical guarantee (success criterion #5) covers
  PRIMARY / HELPER / VAC_CREW / ORIG-FOLDER outputs.** It does
  NOT cover the existing legacy-primary subcontractor file —
  that file disappears (Bug B1) and that's the intended observable
  change. Verification (success criterion #5's hash-history diff)
  must whitelist subcontractor primary tuples as "expected to
  disappear" rather than flag them as regressions.

## Deferred Ideas

- **Generalize `PHASE_1_1_HASH_PRUNE_VERSION` into a reusable
  `HASH_HISTORY_MIGRATION_VERSION` framework.** Phase 1.1 introduces
  the first one-time prune pass; future phases may need similar
  cleanup (e.g., schema drift, retired variant strings). Worth
  consolidating into a single `migrations: list[Callable]` pattern
  once a second use case appears. Premature today.

- **Generalize the per-sheet variant whitelist into a routing
  contract data structure.** Phase 1.1 hardcodes
  `PPP_VARIANT_WHITELIST = {'reduced_sub', 'reduced_sub_helper'}`
  at the call site. If a third target sheet enters the routing
  matrix in a future milestone, lift to a module-level dict
  `{TARGET_SHEET_ID: None, SUBCONTRACTOR_PPP_SHEET_ID:
  {'reduced_sub', 'reduced_sub_helper'}, ...}`. YAGNI for now.

- **Add `attribution_snapshot.last_seen_at` for stale-detection.**
  Bug C's first-write-wins semantics means a row's frozen helper
  is the FIRST helper observed. If the data team's attribution
  snapshot does NOT carry a `last_seen_at` or `last_modified_at`
  column, we can't detect when a row was actually completed
  (vs first seen as a draft entry). This is a future refinement —
  for v1.0 we trust first-write-wins because the operator workflow
  is: row created → helper assigns → completion-checked. The
  "completion-checked" moment is the first time freeze_row sees
  both checkboxes; that's the freeze.

- **Parallelize the Bug C reader.** If per-row attribution_snapshot
  lookups serialize and add measurable latency at scale
  (~550 rows/run, plus helper-row subset), wrap the reader in a
  ThreadPoolExecutor matching the freeze_row pattern. Skip for v1
  because freeze_row writes are already once-per-row in parallel,
  and the read is ONE additional call per helper row (~100 rows
  on a busy week). Re-evaluate if cron runtime regresses.

- **Self-test the data-team RPC contract.** Add a startup-time
  smoke test that calls `lookup_attribution` with a known-good
  (WR, week, row_id) and asserts the response shape. Fails loud
  on schema drift instead of silently falling back to current-helper.
  Worth doing in a follow-up phase if the data team's contract
  proves unstable.
