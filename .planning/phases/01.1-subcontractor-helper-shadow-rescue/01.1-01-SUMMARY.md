# Phase 01.1 Plan 01: Subcontractor Pre-Acceptance Helper-Shadow Rescue Summary

Bug A pre-acceptance rescue branch landed for subcontractor sheets:
helper rows with blank SmartSheet `Units Total Price` now get rescued
via `_SUBCONTRACTOR_RATES.get(cu)['reduced_*_price'] * qty` BEFORE the
`has_price` row-acceptance gate at `generate_weekly_pdfs.py:3714`.
The rescue path mutates `price_val` AND `row_data['Units Total Price']`
in-place (the latter required so the gate's `price_raw not in (None,
"", "$0", ...)` check also passes — same in-place pattern as
`recalculate_row_price` at L1751). When the rescue fires, the row
stamps `row_data['__subcontractor_rescued'] = True` for the Plan
01.1-05 e2e regression test to detect, and emits a diagnostic INFO
log gated on `FILTER_DIAGNOSTICS` + the per-sheet row counter. The
new kill switch `SUBCONTRACTOR_RATE_RECALC_PREACCEPTANCE_ENABLED`
defaults to ON, is pinned in the GitHub Actions workflow, and is
documented in `website/docs/reference/environment.md`.

## Key Decisions Made

- **Stripped local helper over full `_resolve_row_price` second pass**
  (researcher's `01.1-RESEARCH.md` Pattern §A1 recommendation). The
  rescue only needs to confirm a CU has a rate and set `price_val > 0`
  — picking a `variant` at pre-acceptance time would couple Bug A to
  Phase 1's variant semantics unnecessarily. `_resolve_row_price`
  runs downstream inside `generate_excel` at L4887 where actual
  output pricing happens. ~15 lines of new code in a self-contained
  helper vs ~50 lines of refactoring `_resolve_row_price`'s variant
  gating to accept a "rescue mode."
- **Helper defined BEFORE `_resolve_row_price`** (the canonical
  pricing helper at L1467 post-edit) so any future caller in this
  module sees the rescue helper on first read. The plan explicitly
  required this ordering and the executor preserved it.
- **Reduced-rate columns as the pre-acceptance safety floor**, NOT
  new-rate columns. The AEP cutoff (`_AEP_BILLABLE_CUTOFF`) is
  variant-emission gating only — the actual `_AEPBillable` vs
  `_ReducedSub` price selection happens later in `_resolve_row_price`
  at variant-write time. The rescue path doesn't need to (and
  shouldn't) make a cutoff decision at the pre-acceptance gate.
- **In-place mutation of BOTH `price_val` AND
  `row_data['Units Total Price']`** so the `has_price` gate at L3715
  (`(price_raw not in (None, "", "$0", "$0.00", "0", "0.0")) and
  price_val > 0`) passes both clauses. This matches the existing
  `recalculate_row_price` in-place behaviour at L1751 and was added
  as a Rule 2 deviation (see Deviations section below).

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 2 — Critical functionality] Rescue must also mutate
`row_data['Units Total Price']` for `has_price` gate to pass**

- **Found during:** Task 2 implementation while tracing the
  `has_price` recompute at L3715.
- **Issue:** The plan body specified only `price_val = _rescued`
  mutation inside the rescue branch. But the gate at L3715 is
  `(price_raw not in (None, "", "$0", "$0.00", "0", "0.0")) and
  price_val > 0` — BOTH clauses must be true. With `Units Total
  Price=""` (the typical "blank" SmartSheet value), `price_raw=""`
  is in the blank-set, so `has_price` stays False even after
  `price_val` is rescued. The rescue would silently fail to actually
  rescue any row whose raw SmartSheet price is an empty string.
- **Fix:** Added `row_data['Units Total Price'] = _rescued` alongside
  the `price_val = _rescued` mutation. Mirrors the existing
  `recalculate_row_price` in-place pattern at L1751 (which is what
  the analog primary-rate rescue at L3637-3712 relies on).
- **Files modified:** `generate_weekly_pdfs.py` (rescue branch in
  `_fetch_and_process_sheet`)
- **Commit:** `7564ec8`

**2. [Rule 3 — Blocker resolution] Workflow tooling environment
encoding pitfalls**

- **Found during:** Task 3 verification.
- **Issue:** Two encoding pitfalls on the Windows shell: (a)
  `python -c "import yaml; yaml.safe_load(open(...))"` failed with
  `UnicodeDecodeError` on cp1252 default codec (the workflow YAML
  has Unicode comment characters). (b) The `python -c "import
  generate_weekly_pdfs"` introspection failed with
  `UnicodeEncodeError` on the cp1252 stdout because the module's
  startup banner prints emoji. (c) `pytest tests/` initially failed
  with `ModuleNotFoundError: No module named 'generate_weekly_pdfs'`
  because pytest doesn't auto-add CWD to `sys.path`.
- **Fix:** Used `PYTHONIOENCODING=utf-8`, explicit `encoding='utf-8'`
  for `yaml.safe_load(open(..., encoding='utf-8'))`, and
  `PYTHONPATH=.` for pytest. No code changes — environment-only.
- **Files modified:** none
- **Commit:** none
