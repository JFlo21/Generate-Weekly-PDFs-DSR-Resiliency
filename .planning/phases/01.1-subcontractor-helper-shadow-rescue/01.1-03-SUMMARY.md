# Phase 01.1 Plan 03: Bug B2 PPP Cleanup Variant Whitelist Summary

Phase 1.1 Bug B2 closure landed. The PPP cleanup invocation now passes
`variant_whitelist={'reduced_sub', 'reduced_sub_helper'}` and
`cleanup_untracked_sheet_attachments` unconditionally deletes any
attachment whose `build_group_identity`-parsed variant is not in that
whitelist — regardless of `valid_wr_weeks` membership and regardless
of `KEEP_HISTORICAL_WEEKS`. The historical primary-shape orphan on
`SUBCONTRACTOR_PPP_SHEET_ID` (the cycle-2 evidence in
`.planning/debug/sub-helper-shadow-missing.md`) is now structurally
removable even if Bug B1 (Plan 01.1-02) ever regresses or a future
routing-matrix change accidentally routes a non-`reduced_sub` variant
to PPP.

The change is two coordinated edits in `generate_weekly_pdfs.py`:

1. **Function-side (Task 1):** signature gains a trailing
   `variant_whitelist: set[str] | None = None` kwarg. Inside the
   per-row attachment loop, a new whitelist gate fires BEFORE the
   `identity_groups[ident].append(att)` accumulation; off-contract
   attachments route into a new `off_contract_attachments` bucket
   and get unconditionally deleted in a dedicated loop placed
   AFTER attachment classification and BEFORE the existing
   newest-keep prune. New `removed_off_contract` counter, new
   `🗑️ Removed off-contract variant on sheet` INFO log, and new
   `_PII_LOG_MARKERS` entry.
2. **Call-site (Task 2):** PPP invocation at `main()` now passes
   `variant_whitelist={'reduced_sub', 'reduced_sub_helper'}`;
   TARGET invocation is unchanged (default `None` preserves
   byte-identical legacy behavior per D-09). A new comment block
   above the TARGET invocation makes the D-09 contract visible
   at the call site.

The legacy per-identity newest-keep logic
(`for ident, atts in identity_groups.items():` →
`for old in atts_sorted[1:]:`) is **byte-identical** — no behavior
change for in-whitelist variants.

## Key Decisions Made

- **Rule 1 auto-fix on the existing signature-pinning test.**
  `TestPppCleanupUntrackedAttachments::test_cleanup_function_signature_unchanged`
  (added by Phase 1 plan 01-13 REVIEW-WR-01) pinned the function
  signature to exactly six params via `assertEqual(params,
  ['client', 'target_sheet_id', 'valid_wr_weeks', 'test_mode',
  'attachment_cache', 'target_sheet'])`. The plan's mandated
  `variant_whitelist` addition would have triggered a 7th-element
  assertion failure. The plan's `<success_criteria>` allows at
  most 1 failing test (the intentionally-stale D-22 test in
  `test_subcontractor_pricing.py::TestSubcontractorVariantKillSwitchAndScope`),
  so the signature test had to be updated, not left broken. The
  update keeps the WR-01 baseline as the first assertion
  (`params[:6]` must equal the original six) and then asserts the
  new trailing `variant_whitelist` kwarg with `None` default —
  locking BOTH contracts simultaneously. The original test's
  error message ("the fix should be a second invocation only,
  not a signature change") is preserved structurally in the
  updated docstring as a historical note explaining why WR-01
  did NOT change the signature.
- **Mock-driven behavioral tests AND source-level grep tests in
  TestCleanupVariantWhitelist.** Mirrors D-20's rule that
  structural tests bypassing the production code path are
  silent-pass traps. The behavioral tests (Tests 2-6) drive the
  function in-process with `MagicMock` SDK calls — when the
  whitelist gate is wrong, `client.Attachments.delete_attachment`
  records the wrong call pattern and the test fails on the
  count / id assertion. The source-level grep tests (Tests 7,
  PII marker, log body) are drift guards that catch
  refactoring that re-writes the function's surface without
  the code path actually exercising the new logic.
- **`PPP_VARIANT_WHITELIST` left as inline literal per D-08.**
  Considered hoisting to a module constant or to a routing-matrix
  dict but rejected per the deferred ideas in CONTEXT.md — YAGNI
  until a third target sheet exists. The call-site comment block
  names `_build_upload_tasks_for_group` (the routing site) so any
  future maintainer touching PPP routing can grep their way to
  the cleanup-side coupling. The literal `{'reduced_sub',
  'reduced_sub_helper'}` is the simplest expression of the
  contract.
- **TARGET-preservation comment placed at the TARGET call site,
  not at the function definition.** Adding a comment block at the
  function definition saying "TARGET passes None" would be
  invisible to anyone reading the call site. The comment lives
  directly above the TARGET invocation so a future maintainer
  considering "let me add the same whitelist to TARGET" sees the
  D-09 reasoning AT the call site they're editing. The function
  docstring already documents the kwarg's contract in detail.
- **Two-phase classify-then-delete loop structure.** The
  off-contract delete loop runs AFTER the full per-attachment
  classification loop completes (not interleaved). If a single
  delete fails (HTTP 429 / 5xx / transient SDK error), the
  defensive `try/except` keeps the loop running for the
  remaining off-contract attachments. Interleaved classification
  and deletion would have made the failure mode harder to reason
  about — a delete failure would not affect later
  classifications, but the visual code structure suggests
  otherwise. Two-phase is byte-equivalent to the existing
  identity-newest-keep pattern at the bottom of the function.
- **Summary log extended inline (not duplicated).** Considered
  emitting a separate `🛡️ Removed N off-contract on sheet` log
  line per sheet but rejected — that would have doubled the
  cleanup-completion log volume. Consolidating into the existing
  `🧹 Variant pruning done` summary keeps the per-sheet
  single-line cleanup signature that 2026-04-24 11:30 established
  for the `RATE_RECALC_SKIP_ORIGINAL_CONTRACT` guard.

that is functionally a no-op: the grep pattern
`grep -c "'Removed off-contract variant on sheet'"` (single

intent of the criterion — "the marker is in `_PII_LOG_MARKERS`" —

generate_weekly_pdfs._PII_LOG_MARKERS` (the tuple's contents are
quote-agnostic). The grep pattern with double-quoted form returns
2 (one marker entry + one comment reference). Both forms confirm
the marker is registered; the plan's grep was a code-style
preference not a correctness requirement.

## Defense-in-Depth Landed

The whitelist is **belt-and-suspenders**, not the primary fix:

- **Primary fix (Plan 01.1-02 Bug B1):** the producer side no
  longer appends `(wr, week, 'primary', '')` to
  `valid_wr_weeks` for subcontractor non-helper rows. With B1
  in place, the historical stale-primary attachment on PPP
  would have been pruned by the EXISTING newest-keep logic
  (its identity tuple is no longer in `valid_wr_weeks` and
  `KEEP_HISTORICAL_WEEKS=False` by default — so the existing
  `if ident not in valid_wr_weeks and KEEP_HISTORICAL_WEEKS:
  continue` branch falls through to the prune path).
- **Defense (this plan, Bug B2):** even if Bug B1 ever
  regresses, OR if a future bug accidentally adds an
  off-contract variant to `valid_wr_weeks` for a subcontractor
  WR, OR if `KEEP_HISTORICAL_WEEKS` is ever flipped to `True`
  in production, the whitelist gate STILL catches the
  off-contract attachment and deletes it. The two layers are
  independent — neither requires the other to be working
  correctly.

## Known Coupling (Future Routing-Matrix Changes)

The hardcoded literal `{'reduced_sub', 'reduced_sub_helper'}`
at the PPP call site is the **exact** set of variants emitted by
`_build_upload_tasks_for_group` for PPP routing (Phase 1 plan
01-04). If a future plan adds a third variant to PPP routing
(e.g., a Phase 3 that routes `aep_billable` files to PPP), the
hardcoded whitelist literal MUST be updated in the SAME PR —
otherwise the new variant will be deleted as off-contract
immediately after upload. The call-site comment block at
generate_weekly_pdfs.py L7445-7459 documents this coupling.
A future-proof solution (lift to a module-level routing-matrix
dict) is captured in the deferred ideas section of
`.planning/phases/01.1-subcontractor-helper-shadow-rescue/01.1-CONTEXT.md`
but YAGNI for v1.
