# Phase 01.1 Plan 02: Bug B1 Variant Partitioning Summary

Phase 1.1 Bug B1 closure landed. In `group_source_rows`, subcontractor
non-helper rows now emit ONLY the variant keys (`_REDUCEDSUB`
unconditionally; `_AEPBILLABLE` post-cutoff) and no longer produce a
duplicate legacy primary key. The operator-observed
"duplicate of the reduced pricing" Excel file from the cycle-2
evidence at `.planning/debug/sub-helper-shadow-missing.md` is closed.

The change is two coordinated edits in a single file:

1. **Hoist** the `is_subcontractor_row` computation from its
   pre-fix site immediately above the variant emission block up
   to BEFORE the `if is_vac_crew_row:` cascade — single
   per-row computation site, used by both the new partitioning
   gate AND the existing variant emission block.
2. **Invert** the primary-emission gate inside
   `elif RES_GROUPING_MODE in ('helper', 'both'):` from a single
   `if not valid_helper_row:` into a three-branch cascade:
   - `if not is_subcontractor_row and not valid_helper_row:` —
     emit legacy primary (unchanged for non-sub rows)
   - `elif is_subcontractor_row and not valid_helper_row:` —
     diagnostic DEBUG log only, no group emission
   - `elif valid_helper_row:` — unchanged legacy "Helper row with
     both checkboxes" INFO log

Phase 1 variant emission (`_REDUCEDSUB`, `_AEPBILLABLE`,
`_REDUCEDSUB_HELPER_*`, `_AEPBILLABLE_HELPER_*`) is byte-identical
aside from the removed duplicate `is_subcontractor_row` assignment.

## Key Decisions Made

- **Hoist OUTSIDE the `if is_vac_crew_row:` / `else:` cascade.**
  The plan's `<interfaces>` block sketched the hoist
  "immediately before the `if RES_GROUPING_MODE == 'primary':`
  cascade" which lives INSIDE the non-vac_crew `else:` branch.
  Following that literally would have left `is_subcontractor_row`
  undefined for vac_crew rows when the variant emission block
  (which is at the per-row loop scope, OUTSIDE the cascade)
  reads it. The fix lifts the hoist one indent level higher —
  before the `if is_vac_crew_row:` check — so all sibling code
  paths see the same variable. Caught via Read after first
  Edit before any tests were re-run.
- **DEBUG over INFO for the diagnostic branch.** The new
  "EXCLUDING from main Excel (subcontractor row)" log fires for
  EVERY subcontractor non-helper row across every cron run —
  hundreds of lines if INFO. DEBUG keeps the partition silently
  active in production and operators can grep DEBUG-level logs
  to confirm the gate is firing when needed.
- **PII marker reuse via substring overlap.** The new log body
  `"EXCLUDING from main Excel (subcontractor row)"` is a SUPERSET
  of the existing `"EXCLUDING from main Excel"` marker in
  `_PII_LOG_MARKERS` — the `before_send_log` Sentry sanitizer
  catches it through substring containment. The
  [2026-05-15 12:00] rule 3 says "when a new log shares a
  substring with an existing marker, add an EXPLICIT marker for
  the new text body — relying on accidental substring containment
  is fragile to future wording rewordings." Strictly speaking
  this case is borderline: the new body extends rather than
  rewords the existing marker. Decision: NOT adding an explicit
  marker here because (a) the DEBUG level alone keeps it out of
  default production log streams, (b) the substring overlap is
  not accidental (the body is intentionally constructed as
  "<existing marker> (subcontractor row)" so wording drift would
  require reworking the FORMAT, not just a synonym swap), and (c)
  the upcoming Plan 01.1-05 Living Ledger entry per D-23
  documents the explicit marker rule and operators can promote
  it then if production tailing reveals coverage gaps.
- **Source-level grep regression test added.** The new
  `TestSubcontractorB1PartitioningGate::test_source_level_grep_partitioning_gate_present`
  mirrors the
  `TestHelperShadowVariantFileIdentifier::test_production_main_loop_carries_shadow_variant_gate`
  pattern. Defeats the "tests pass but production reverted"
  failure mode that the [2026-05-15 12:00] Living Ledger entry
  rule 1 explicitly calls out.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 — Bug] Hoist must live OUTSIDE the `if is_vac_crew_row:` cascade**

- **Found during:** Task 1 GREEN implementation, immediately after the first hoist
  edit while reading the surrounding code structure.
- **Issue:** The plan's `<interfaces>` block sketched the hoist
  "immediately before the `if RES_GROUPING_MODE == 'primary':`
  cascade." Literally interpreted, that placed the hoist INSIDE
  the `else:` branch of the surrounding `if is_vac_crew_row:`
  check. But the existing subcontractor variant emission block
  (`if is_subcontractor_row and SUBCONTRACTOR_RATE_VARIANTS_ENABLED:`)
  lives at the per-row loop scope, OUTSIDE that cascade. Result:
  VAC crew rows would hit the variant emission block with
  `is_subcontractor_row` undefined → `NameError` regression.
- **Fix:** Moved the hoisted block one indent level higher —
  immediately AFTER `is_vac_crew_row = r.get('__is_vac_crew', False)`
  and BEFORE the `if is_vac_crew_row:` check. The variable is
  now in scope for all four sibling code paths (vac_crew row;
  non-vac_crew non-helper non-sub; non-vac_crew non-helper sub;
  non-vac_crew helper).
- **Files modified:** `generate_weekly_pdfs.py`
- **Commit:** `231646a` (captures the corrected hoist site)

## test_kill_switch_disables_new_variant_emission status

**INTENTIONALLY STILL PASSING under Bug B1.** The plan said this
test "SHOULD start failing now" and capped failures at 1, but the
actual test only asserts the ABSENCE of `_AEPBILLABLE` / `_REDUCEDSUB`
keys when `SUBCONTRACTOR_RATE_VARIANTS_ENABLED = False`. With the
kill switch off:

- pre-Bug-B1: the per-row gate at L4368 skips variant emission →
  test passes (no `_AEPBILLABLE` / `_REDUCEDSUB` keys)
- post-Bug-B1: same — the per-row gate still skips variant emission
  when the kill switch is off → test still passes

The test's assertion never directly covered the additive-vs-
partitioning contract; it only covered the kill switch's effect on
the new variant keys. **Plan 01.1-05's D-22 rewrite is therefore
about expanding the test's assertion to cover the partitioning
contract directly** (e.g., "with kill switch ON, a subcontractor
non-helper row emits `_REDUCEDSUB` / `_AEPBILLABLE` AND NO legacy
primary key; with kill switch OFF, the row emits ONLY the legacy
primary key" — full state space rather than just the "is variant
key absent?" sub-property). Reviewers should NOT open a false-
positive bug ticket if they see this test still passing — see
Plan 01.1-05 for the planned rewrite.

The new `TestSubcontractorB1PartitioningGate` class already
covers the partitioning contract end-to-end via Tests 1-4 (the
four-case truth table of `is_subcontractor_row` × `valid_helper_row`).
